# Office Unify Project Charter

## 목적
이 프로젝트의 목적은 기존 두 프로젝트를 개별 서비스로 운영하지 않고,
`dev_support`를 최종 제품 본체로 삼아 `ai_office`의 핵심 기능과 로직을 점진적으로 통합하는 것이다.

## 최상위 원칙
1. 최종 제품의 중심은 `dev_support`이다.
2. `ai_office`는 기능 원천이자 참조 소스이며, 직접 수정은 최소화한다.
3. 통합의 목표는 **dev_support 중심 통합 + ai_office 서버 로직 흡수**이다.
4. 기존 `dev_support`의 기능(업무 도우미: flow/sql/ts 생성 등)은 깨지면 안 된다.
5. Discord UI/interaction 의존성은 웹 UI 또는 adapter 계층으로 치환한다.
6. 순수 비즈니스 로직은 가능한 한 재사용한다.
7. 민감정보(API Key, Supabase service role key 등)는 반드시 서버 전용으로 유지한다.
8. 클라이언트에는 최소한의 안전한 데이터만 노출한다.
9. 큰 작업은 항상 **분석 → 설계 → 구현 → 검증 → 문서화** 순서로 진행한다.
10. 문서와 검증은 구현과 동일한 수준의 필수 산출물이다.

## 제품 방향
- 최종 통합 제품은 하나의 웹 앱이다.
- 웹 앱 안에서 `AI Office`와 `업무 도우미`를 함께 제공한다.
- `ai_office`의 Discord 기반 입출력은 제거하거나 legacy adapter로 격리한다.
- 실제 분석, 판단, 조회, 데이터 처리 로직은 웹 앱의 서버 계층으로 이동한다.

## 작업 대상 우선순위
1. `dev_support` 분석 및 확장
2. `ai_office` 기능/로직 분해 및 참조
3. 점진적 마이그레이션
4. 필요 시 adapter/legacy 유지

## 금지 사항
- 승인 없는 대규모 구조 파괴
- 기존 `dev_support` 핵심 기능 손상
- 민감정보의 클라이언트 노출
- Discord 결합 로직의 무분별한 직접 이식
- 검증 없는 완료 선언

## 모든 작업의 기본 산출물
모든 주요 작업은 아래 형식으로 결과를 정리해야 한다.
- 설계 요약
- 변경 파일 목록
- 핵심 변경 이유
- 검증 방법
- 남은 리스크 / 다음 단계

## 하네스 엔지니어링 운영 원칙
이 프로젝트는 프롬프트 하나에 의존하지 않는다.
항상 다음을 함께 관리한다.
- 프로젝트 규칙
- 설계 문서
- 단계별 계획
- 검증 기준
- 보고 형식
- 리스크 기록

Cursor는 위 원칙을 따르는 작업 보조자이며,
모든 구현은 이 헌장을 위반하지 않는 범위에서만 진행한다.
