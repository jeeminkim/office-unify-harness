# Office Unify Architecture Boundaries

## 목적
이 문서는 `dev_support`와 `ai_office`를 하나의 웹 서비스로 통합하는 과정에서
아키텍처 경계를 명확히 정의하기 위한 기준 문서다.

통합 과정에서 가장 중요한 것은
기존 로직을 무조건 옮기는 것이 아니라,
**웹 환경에 맞는 안전한 계층 구조로 재배치하는 것**이다.

---

## 1. 최상위 구조 원칙

1. 최종 제품의 중심 구조는 `dev_support` 기준으로 설계한다.
2. `ai_office`는 기능 원천이며, 구조의 기준점이 아니다.
3. UI는 웹 기준으로 재설계하고, Discord UI 개념은 직접 이식하지 않는다.
4. 가능한 한 순수 비즈니스 로직만 추출하여 서버 계층으로 이동한다.
5. 각 계층은 역할이 명확해야 하며, UI/서비스/저장소/외부 연동을 혼합하지 않는다.

---

## 2. 클라이언트와 서버 경계

### 클라이언트가 담당하는 것
- 화면 렌더링
- 사용자 입력 수집
- 상태 표시
- 결과 시각화
- 사용자 액션 트리거
- 안전한 API 호출

### 서버가 담당하는 것
- LLM 호출
- Supabase 접근
- 민감한 환경변수 사용
- 투자/분석/판단 로직 실행
- 데이터 정규화/검증
- 외부 API 연동
- 감사/운영 로그 처리

### 절대 금지
- 클라이언트에서 API Key 직접 사용
- 클라이언트에서 Supabase service role 사용
- 클라이언트에서 민감한 판단 로직 직접 실행
- 서버 전용 로직을 브라우저 코드에 혼합

---

## 3. ai_office 코드 분류 원칙

`ai_office` 코드는 반드시 아래 4가지로 분류해서 본다.

### A. 순수 로직
예:
- 분석 규칙
- 데이터 가공
- 의사결정 계산
- provider 선택 로직
- 결과 조합 로직

이 범주의 코드는 우선적으로 재사용 후보로 본다.

### B. 인프라/외부 연동 로직
예:
- OpenAI/Gemini 호출
- Supabase 접근
- 외부 시세 조회
- 환경변수 의존 로직

이 범주의 코드는 서버 계층으로 이동 후보로 본다.

### C. Discord 결합 로직
예:
- interaction 처리
- panel restore
- ephemeral reply
- customId 기반 라우팅
- Discord message formatting

이 범주의 코드는 직접 이식하지 않는다.
필요 시 adapter 또는 legacy 계층으로 격리한다.

### D. 운영 전용 로직
예:
- 재기동 알림
- 운영 패널 제어
- Discord 관리자 액션
- PM2 연동 흐름

이 범주의 코드는 웹 전환 가치가 높은 것만 읽기 중심으로 재설계하고,
기존 Discord 운영 기능은 즉시 제거하지 않는다.

---

## 4. Discord 개념의 웹 치환 원칙

기존 Discord 개념은 웹에서 그대로 복제하지 않는다.
아래처럼 의미 중심으로 치환한다.

- Discord button → 웹 버튼 / 액션 UI
- modal → 웹 폼 / 다이얼로그
- ephemeral reply → 사용자별 상태/결과 패널
- interaction routing → 웹 라우트/API 핸들러
- panel restore → 웹 대시보드 초기 상태/복구 로직
- admin-only control → 인증된 운영자 페이지

즉, Discord의 기술적 구조를 옮기는 것이 아니라,
사용자 경험과 기능 목적만 웹에 맞게 재설계한다.

---

## 5. 서버 구조 원칙

통합 이후 서버 구조는 가능한 한 역할별로 분리한다.

예시:
- `src/server/ai-office/contracts/`
- `src/server/ai-office/providers/`
- `src/server/ai-office/services/`
- `src/server/ai-office/repositories/`
- `src/server/ai-office/utils/`
- `app/api/ai-office/*`

### 계층 역할
- contracts: 타입, 인터페이스, DTO
- providers: OpenAI/Gemini 등 LLM provider 추상화
- services: 핵심 비즈니스 로직
- repositories: DB 접근
- utils: 순수 공통 유틸
- api: 웹 요청 진입점

### 금지
- API route에서 모든 로직을 직접 처리
- DB 접근과 UI 응답 포맷을 한 함수에 혼합
- provider별 분기 로직을 전체 코드에 흩뿌리기

---

## 6. 상태 관리 원칙

상태는 아래처럼 구분한다.

### 클라이언트 상태
- UI 열림/닫힘
- 선택 탭
- 로딩 상태
- 임시 입력값
- 일회성 결과 표시

### 서버 상태
- 포트폴리오 데이터
- AI 토론 결과
- 트렌드 분석 결과
- 운영 상태
- 사용자별 저장 데이터
- 이력/로그/판단 결과

### 원칙
- 장기 보관 데이터는 서버 기준으로 관리한다.
- MVP 단계라도 투자/운영 데이터는 localStorage 중심 구조에 과도하게 의존하지 않는다.
- 캐시는 가능하지만, 진실의 원천(source of truth)은 서버여야 한다.

---

## 7. 보안 경계 원칙

1. 모든 민감정보는 서버 전용 환경변수로 유지한다.
2. 외부 API Key는 클라이언트 번들에 포함되면 안 된다.
3. Supabase service role key는 서버에서만 사용한다.
4. 사용자에게 필요한 최소 데이터만 응답한다.
5. 로그에도 민감정보를 남기지 않는다.
6. 디버깅 편의보다 보안 경계를 우선한다.

---

## 8. 마이그레이션 경계 원칙

1. 한 번에 모든 기능을 옮기지 않는다.
2. 순수 로직 → 서버 계층 → 웹 UI 순서로 이동한다.
3. 운영 기능은 읽기 중심 대시보드부터 시작한다.
4. Discord 전용 기능은 필요 시 legacy adapter로 남긴다.
5. 기존 기능 안정성이 새로운 구조 완성도보다 우선이다.

---

## 9. 판단 기준

새 코드를 추가하거나 기존 코드를 옮길 때는 항상 아래를 자문한다.

- 이 로직은 UI인가, 서버인가?
- 이 로직은 Discord에 묶여 있는가?
- 이 로직은 순수 로직으로 추출 가능한가?
- 민감정보가 섞여 있는가?
- 웹 구조로 재설계해야 하는가, 아니면 재사용 가능한가?
- 지금 당장 옮길 가치가 있는가?

위 질문에 답하지 못하면, 바로 구현하지 말고 먼저 분류와 설계를 수행한다.

---

## 10. 이 문서의 역할

이 문서는 통합 구현 시의 기술적 경계선이다.
Cursor 또는 다른 AI 도구는 이 문서를 기준으로
어떤 코드를 재사용하고, 어떤 코드를 격리하고,
어떤 구조를 새로 설계해야 하는지를 판단해야 한다.
