# Office Unify Migration Workflow

## 목적
이 문서는 `ai_office`와 `dev_support`를 하나의 웹 서비스로 통합하는 과정에서
작업 순서, 변경 단위, 검증 흐름을 일관되게 유지하기 위한 운영 규칙이다.

이 프로젝트의 핵심은 빠른 대규모 이식이 아니라,
**안전한 점진적 통합**이다.

---

## 1. 기본 작업 순서

모든 주요 작업은 반드시 아래 순서를 따른다.

1. 분석
2. 설계
3. 작은 단위 구현
4. 검증
5. 문서화
6. 결과 보고

이 순서를 건너뛰지 않는다.

---

## 2. 분석 단계 규칙

분석 단계에서는 코드를 크게 수정하지 않는다.

분석 단계의 목표:
- 현재 구조 파악
- 관련 파일 식별
- 순수 로직 / Discord 결합 로직 분류
- 재사용 가능성 판단
- 리스크 도출
- 예상 변경 파일 식별

분석 단계에서 반드시 포함할 것:
- 어떤 디렉토리/파일이 관련되는지
- 무엇을 바로 옮길 수 있는지
- 무엇을 격리해야 하는지
- 어떤 부분이 웹 구조로 재설계되어야 하는지

### 분석 단계 금지
- 대규모 파일 수정
- 설계 없는 직접 구현
- 추정 기반 완료 선언

---

## 3. 설계 단계 규칙

설계 단계에서는 구현 전에 구조를 먼저 고정한다.

설계 단계의 목표:
- 대상 기능 범위 확정
- 계층 구조 정의
- API 경로 초안 작성
- 상태 관리 방식 제안
- 보안 경계 확인
- 예상 변경 파일 목록 구체화

설계 단계의 결과물 예시:
- 폴더 구조 제안
- 데이터 흐름 설명
- API route 설계안
- UI 전환 흐름
- 마이그레이션 순서

### 설계 단계 원칙
- 설계는 구현 가능한 수준으로 구체적이어야 한다.
- 한 번에 너무 넓은 범위를 잡지 않는다.
- 1차 구현 범위는 MVP 기준으로 제한한다.

---

## 4. 구현 단계 규칙

구현은 항상 작은 단위로 나눈다.

### 구현 단위 원칙
- 한 번에 하나의 주제만 다룬다.
- 변경은 논리적 묶음으로 나눈다.
- 큰 구조 변경보다 점진적 추가를 우선한다.
- 기존 기능을 먼저 보호하고 새 구조를 덧붙인다.

### 권장 구현 순서
1. 공통 구조/폴더 골격
2. 타입/contracts 정리
3. provider/repository/service 분리
4. API route 추가
5. 최소 UI 연결
6. 결과 표시 개선
7. 문서 갱신

### 구현 단계 금지
- 여러 관심사를 한 번에 섞기
- UI, DB, LLM, 라우팅을 한 파일에 몰아넣기
- 기존 동작을 깨뜨리는 일괄 교체
- 검증 없는 “작동할 것 같음” 판단

---

## 5. 변경 크기 제어 원칙

이 프로젝트에서는 “한 번에 크게 바꾸기”를 지양한다.

### 항상 우선해야 하는 것
- 작은 범위
- 명확한 목적
- 쉬운 검토
- 쉬운 롤백
- 쉬운 검증

### 이상적인 변경 단위 예시
- 메뉴/내비게이션 추가
- 서버 폴더 골격 추가
- 포트폴리오 조회 API 1개 추가
- AI 토론 서비스 추출
- 트렌드 분석 실행 흐름 1개 연결

### 피해야 할 변경 예시
- 전체 라우팅 전면 교체
- 모든 Discord 로직 일괄 이전
- UI/서버/DB 구조 동시 전면 개편

---

## 6. 마이그레이션 우선순위

통합 작업은 아래 우선순위를 따른다.

### 1순위
기존 `dev_support` 기능 보호

### 2순위
`ai_office`의 순수 로직 분리 및 재사용 후보 식별

### 3순위
웹 전환 가치가 높은 핵심 기능 흡수
- 포트폴리오 조회
- AI 토론 실행
- 트렌드 분석 실행

### 4순위
운영 기능의 읽기 중심 웹 전환

### 5순위
Discord 전용 legacy 정리 또는 adapter 축소

---

## 7. 검증 단계 규칙

구현 후에는 반드시 검증을 수행한다.

검증 항목 예시:
- lint 통과
- build 통과
- typecheck 통과
- 핵심 기능 smoke test
- 기존 업무 도우미 기능 정상 동작 확인
- 신규 기능 최소 흐름 확인

### 검증 원칙
- 검증 없는 완료 선언 금지
- 오류가 있으면 원인과 범위를 명시
- “일부만 완료”라면 그 상태를 정직하게 보고
- 검증 결과는 산출물에 포함

---

## 8. 문서화 단계 규칙

코드만 바꾸고 끝내지 않는다.

변경 시 함께 점검할 문서:
- README.md
- 관련 docs/*.md
- 아키텍처 설명 문서
- 마이그레이션 계획 문서
- 체크리스트/리스크 로그

### 문서화 원칙
- 구조 변경은 반드시 문서에 반영
- 새 API/폴더 구조는 최소한의 설명 남기기
- 다음 작업자가 이어받을 수 있도록 기록

---

## 9. 결과 보고 형식

모든 주요 작업 결과는 아래 형식을 따른다.

1. 설계 요약
2. 변경 파일 목록
3. 핵심 변경 이유
4. 검증 방법
5. 남은 리스크 / 다음 단계

필요하면 추가로 아래를 포함할 수 있다.
- 예상 영향 범위
- 롤백 포인트
- 후속 권장 작업

---

## 10. 작업 중 의사결정 규칙

구현 중 판단이 필요한 경우 아래 기준으로 결정한다.

1. 기존 기능 안정성이 가장 우선이다.
2. 구조적 일관성이 단기 편의보다 우선이다.
3. 민감정보 보호가 개발 편의보다 우선이다.
4. 점진적 통합이 일괄 이전보다 우선이다.
5. 검증 가능한 변경이 추상적 개선보다 우선이다.

---

## 11. Cursor 작업 방식 규칙

Cursor 또는 다른 AI 도구는 아래 방식으로 일해야 한다.

- 먼저 관련 파일을 찾는다.
- 바로 수정하지 말고 분석/설계를 제시한다.
- 변경 범위를 명확히 한다.
- 한 번에 작은 단위만 구현한다.
- 구현 후 검증 명령을 제안한다.
- 문서 갱신 필요 여부를 함께 판단한다.

### 권장 프롬프트 흐름
- 분석 전용 프롬프트
- 설계 전용 프롬프트
- 구현 전용 프롬프트
- 검증/문서화 프롬프트

---

## 12. 이 문서의 역할

이 문서는 통합 프로젝트의 실행 순서를 고정하는 운영 하네스다.

이 문서를 통해
- 성급한 대규모 변경을 막고
- 작업 단위를 작게 유지하며
- 분석/설계/구현/검증/문서화가 한 흐름으로 이어지도록 한다.

모든 구현은 이 워크플로를 따른다.
