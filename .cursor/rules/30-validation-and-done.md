# Office Unify Validation and Definition of Done

## 목적
이 문서는 `office-unify` 통합 프로젝트에서
어떤 검증을 거쳐야 작업이 완료된 것으로 간주되는지 정의한다.

이 프로젝트에서는 “코드를 수정했다”는 사실만으로 완료를 선언하지 않는다.
완료란 항상 **검증 가능성, 문서 반영, 리스크 공개**를 포함해야 한다.

---

## 1. 완료의 기본 정의

작업은 아래 조건을 충족할 때만 완료로 간주한다.

1. 작업 목적이 충족되었다.
2. 변경 범위가 명확히 보고되었다.
3. 가능한 검증이 수행되었다.
4. 기존 핵심 기능을 해치지 않았는지 점검했다.
5. 필요한 문서가 갱신되었다.
6. 남은 리스크와 후속 과제가 정리되었다.

---

## 2. 완료 선언 전에 반드시 확인할 것

완료 선언 전에는 아래를 반드시 스스로 점검한다.

- 이번 작업의 목표가 무엇이었는가?
- 실제로 어떤 파일을 변경했는가?
- 변경이 기존 기능에 어떤 영향을 주는가?
- 최소한 어떤 검증을 수행했는가?
- 검증하지 못한 부분은 무엇인가?
- 문서를 갱신해야 하는가?
- 남은 리스크는 무엇인가?

위 항목에 답하지 못하면 완료 선언을 하지 않는다.

---

## 3. 검증의 기본 원칙

검증은 가능한 범위 내에서 반드시 수행한다.

### 기본 원칙
1. 검증 없는 완료 선언 금지
2. “아마 될 것 같다”는 표현으로 완료 처리 금지
3. 일부만 검증했으면 그 사실을 명확히 공개
4. 실패한 검증은 숨기지 않고 보고
5. 검증 결과는 산출물에 포함

---

## 4. 권장 검증 항목

프로젝트 상황에 따라 아래 검증을 수행한다.

### 공통 검증
- lint
- build
- typecheck
- import/route 오류 여부 확인
- 환경변수 의존 구간 점검
- 문서 링크/경로 점검

### 기능 검증
- 기존 `dev_support` 핵심 기능 정상 동작 여부
- 신규 UI 진입 가능 여부
- 신규 API 호출 흐름 확인
- 서버 계층 연결 확인
- 예외/에러 처리 흐름 점검

### 통합 검증
- 기존 기능과 신규 기능 간 충돌 여부
- 클라이언트/서버 경계 위반 여부
- 민감정보 노출 여부
- legacy adapter와 신규 구조 간 충돌 여부

---

## 5. 최소 검증 기준

작업 종류별 최소 검증 기준은 아래와 같다.

### A. 문서 작업
- 문서 구조 확인
- 경로/파일명 확인
- 기존 원칙과 충돌 여부 확인

### B. UI 작업
- 빌드 가능 여부
- 라우트 진입 가능 여부
- 주요 화면 렌더링 여부
- 기존 화면 영향 여부

### C. 서버 작업
- 타입/빌드 확인
- API 경로 연결 확인
- 민감정보 서버 전용 유지 여부
- 예외 처리 기본 흐름 확인

### D. 구조 리팩토링
- import 경로 정합성 확인
- 기존 기능 회귀 위험 점검
- 폴더/계층 규칙 위반 여부 확인
- 문서 갱신 여부 확인

---

## 6. 검증 실패 시 원칙

검증이 실패하면 작업을 완료로 선언하지 않는다.

대신 아래 중 하나로 보고한다.

1. 부분 완료
2. 구조 초안만 반영
3. 검증 실패로 보류
4. 추가 수정 필요
5. 환경 의존으로 미확인

### 반드시 함께 적을 것
- 실패한 검증 항목
- 원인 추정
- 영향 범위
- 다음 수정 제안

---

## 7. 문서 반영 기준

아래 경우에는 문서 반영이 필요하다.

- 폴더 구조가 바뀐 경우
- 새 API가 추가된 경우
- 아키텍처 경계가 바뀐 경우
- 마이그레이션 단계가 진척된 경우
- 운영 방식이 바뀐 경우
- 새로운 규칙/제약이 생긴 경우

### 반영 대상 예시
- README.md
- docs/*.md
- .cursor/plans/*.md
- 체크리스트
- 리스크 로그

---

## 8. 모든 주요 작업의 결과 보고 형식

모든 주요 작업 결과는 아래 형식을 따른다.

### 1. 설계 요약
- 무엇을 왜 바꿨는지
- 어떤 원칙을 적용했는지

### 2. 변경 파일 목록
- 실제 변경 파일
- 신규 파일 / 수정 파일 구분 가능하면 표시

### 3. 핵심 변경 이유
- 구조적 이유
- 기능적 이유
- 보안/운영 이유

### 4. 검증 방법
- 실행한 명령
- 확인한 화면/흐름
- 수행하지 못한 검증

### 5. 남은 리스크 / 다음 단계
- 현재 남아 있는 문제
- 추후 이어서 할 작업
- 주의사항

---

## 9. Done 과 Partial Done 구분

### Done
아래를 만족하면 Done 으로 본다.
- 목표 달성
- 검증 수행
- 결과 보고 완료
- 문서 반영 완료
- 리스크 공개 완료

### Partial Done
아래 중 하나라도 해당하면 Partial Done 으로 본다.
- 구현은 했지만 검증 미완료
- 설계만 완료
- 구조 골격만 반영
- 일부 환경에서만 확인
- 후속 작업이 필수

Partial Done 인 경우 Done 처럼 표현하지 않는다.

---

## 10. 금지 표현

아래 표현은 근거 없이 사용하지 않는다.

- 완료되었습니다
- 정상 동작합니다
- 문제 없습니다
- 모두 반영되었습니다
- 배포 가능합니다

위 표현은 실제 검증 근거가 있을 때만 사용한다.

---

## 11. Cursor 작업 결과의 필수 태도

Cursor 또는 다른 AI 도구는 결과를 보고할 때 아래 태도를 지켜야 한다.

- 검증한 것과 검증하지 못한 것을 구분한다.
- 확실한 것과 추정인 것을 구분한다.
- 남은 리스크를 숨기지 않는다.
- 문서 반영 필요성을 함께 판단한다.
- 사용자가 다음 액션을 선택할 수 있게 상태를 명확히 보고한다.

---

## 12. 이 문서의 역할

이 문서는 통합 프로젝트에서
“언제 완료라고 부를 수 있는가”를 정의하는 품질 하네스다.

이 문서를 통해
- 성급한 완료 선언을 막고
- 검증 가능한 산출물을 만들며
- 코드 변경과 문서화, 리스크 관리가 함께 가도록 한다.
