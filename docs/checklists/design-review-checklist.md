# Office Unify Design Review Checklist

## 문서 목적
이 문서는 `office-unify` 통합 프로젝트에서
설계안이 실제 구현으로 넘어갈 만큼 충분히 정리되었는지 검토하기 위한 체크리스트다.

이 체크리스트의 목적은 다음과 같다.

- 설계 범위 과확장 방지
- 구조적 모호성 조기 식별
- 구현 전 리스크 드러내기
- 기존 기능 보호 관점 확인
- 클라이언트/서버/보안 경계 검토
- 검증 및 문서화 계획 포함 여부 확인

이 문서는 구현을 대신하지 않는다.
이 문서는 “설계가 구현 가능한 수준인가”를 빠르게 점검하는 리뷰 도구다.

---

# 1. 사용 시점

이 체크리스트는 아래 상황에서 사용한다.

- 분석 이후 설계안 초안이 나왔을 때
- `.cursor/plans/office-unify-master-plan.md`를 채운 뒤
- 특정 기능 이관 설계가 끝났을 때
- 실제 구현 시작 직전
- 범위가 커져서 설계 재검토가 필요할 때

원칙은 아래와 같다.

1. 설계가 멋져 보여도 체크리스트를 통과하지 않으면 바로 구현하지 않는다.
2. 불확실한 항목은 체크하지 말고 리스크나 추가 확인 대상으로 남긴다.
3. 범위가 크거나 모호하면 구현보다 설계 축소를 우선 검토한다.
4. 기존 기능 보호와 보안 경계는 설계 품질의 핵심 기준으로 본다.

---

# 2. 설계 기본 적합성 체크리스트

## 2.1 현재 단계 적합성
- [ ] 현재 문서는 구현안이 아니라 설계안으로 정리되어 있다.
- [ ] 분석 결과를 바탕으로 설계가 도출되었다.
- [ ] 지금 단계에서 필요한 설계 수준이 과도하지 않다.
- [ ] 구현 없이도 설계 의도를 설명할 수 있다.
- [ ] 설계가 실제 다음 단계 구현과 연결된다.

## 2.2 설계 목표 명확성
- [ ] 이번 설계의 목표를 한 문장으로 설명할 수 있다.
- [ ] 왜 지금 이 설계를 하는지 설명 가능하다.
- [ ] 이번 설계가 전체 통합에서 어느 위치인지 설명 가능하다.
- [ ] 포함 범위와 제외 범위가 구분되어 있다.
- [ ] 이번 설계에서 하지 않을 것이 정리되어 있다.

---

# 3. 프로젝트 원칙 정합성 체크리스트

## 3.1 제품 중심축 정합성
- [ ] 최종 제품 본체가 `dev_support`라는 점을 유지하고 있다.
- [ ] `ai_office`는 참조 원본으로 다루고 있다.
- [ ] 구조의 중심이 `ai_office` 쪽으로 기울지 않는다.
- [ ] 기존 `dev_support`의 사용자 경험을 무시하지 않는다.

## 3.2 기존 기능 보호
- [ ] 기존 업무 도우미 핵심 기능 보호 계획이 있다.
- [ ] 설계가 기존 기능 전면 교체를 전제로 하지 않는다.
- [ ] 회귀 위험이 높은 변경은 작은 단계로 쪼개져 있다.
- [ ] 기존 기능 영향 범위를 최소한으로 줄이려는 구조다.

## 3.3 Discord 직접 이식 방지
- [ ] Discord interaction 구조를 그대로 옮기지 않는다.
- [ ] Discord button/modal/ephemeral 개념을 웹에 맞게 재해석했다.
- [ ] customId 기반 흐름을 그대로 유지하려 하지 않는다.
- [ ] Discord 결합 로직은 분리/격리/adapter 후보로 다룬다.

---

# 4. 구조/아키텍처 체크리스트

## 4.1 구조 명확성
- [ ] 설계안에 최상위 구조가 설명되어 있다.
- [ ] 폴더 구조 제안이 존재한다.
- [ ] 새 구조와 기존 구조의 관계가 설명되어 있다.
- [ ] 어떤 계층이 새로 생기는지 명확하다.
- [ ] 구조가 다음 단계 확장성을 해치지 않는다.

## 4.2 계층 분리
- [ ] UI, API, service, repository, provider, contract 역할이 구분되어 있다.
- [ ] API route에서 모든 로직을 처리하려 하지 않는다.
- [ ] 서비스 로직과 DB 접근 로직을 분리할 계획이 있다.
- [ ] 외부 연동은 provider 또는 유사 계층으로 묶을 계획이다.
- [ ] 공통 유틸과 도메인 로직이 혼합되지 않도록 고려했다.

## 4.3 클라이언트/서버 경계
- [ ] 화면/입력/시각화는 클라이언트 역할로 정의했다.
- [ ] 민감정보/외부 API/판단 로직은 서버 역할로 정의했다.
- [ ] 클라이언트에 두면 안 되는 로직이 구분되어 있다.
- [ ] source of truth를 서버 기준으로 둘 계획이 있다.
- [ ] localStorage 의존 범위를 제한하고 있다.

---

# 5. 기능 이관 설계 체크리스트

## 5.1 순수 로직 분류
- [ ] 순수 로직 후보가 식별되어 있다.
- [ ] 왜 순수 로직인지 설명 가능하다.
- [ ] 추출 후 재사용 가능한지 판단이 있다.
- [ ] 웹/Discord와 무관한 로직이 우선 후보로 잡혀 있다.

## 5.2 서버 전용 로직 분류
- [ ] 외부 API 호출은 서버 전용으로 본다.
- [ ] Supabase 접근을 서버 기준으로 설계한다.
- [ ] 민감정보를 다루는 로직이 서버에 남는다.
- [ ] provider/service/repository 분리 필요성이 정리되어 있다.

## 5.3 Discord 결합 로직 분류
- [ ] Discord 결합 로직 목록이 정리되어 있다.
- [ ] 직접 이식 금지 대상이 분명하다.
- [ ] 웹 치환 방향이 설명되어 있다.
- [ ] legacy adapter 필요 여부가 검토되었다.

## 5.4 운영 기능 분류
- [ ] 운영 기능을 일반 사용자 기능과 구분했다.
- [ ] 읽기 중심 웹 전환 가능성을 검토했다.
- [ ] 기존 운영 흐름을 당장 제거하지 않는 방안을 고려했다.
- [ ] 운영 공백 위험을 점검했다.

---

# 6. 라우팅 / IA 체크리스트

## 6.1 라우트 구조
- [ ] 주요 라우트 제안이 있다.
- [ ] 기존 라우트와 신규 라우트의 공존 방식이 설명되어 있다.
- [ ] 상위 진입점(`/`, `/ai-office`, `/work-helper` 등)이 명확하다.
- [ ] 신규 라우트가 기존 진입 흐름을 해치지 않도록 고려했다.

## 6.2 정보 구조
- [ ] 통합 홈/AI Office/업무 도우미의 관계가 설명되어 있다.
- [ ] 공통 내비게이션 방향이 정리되어 있다.
- [ ] 상태 배지/결과 패널/에러 상태 등 공통 UI 개념이 정리되어 있다.
- [ ] 사용자 입장에서 기능 구분이 자연스러운지 검토했다.

---

# 7. 상태 관리 / API 체크리스트

## 7.1 상태 관리
- [ ] 클라이언트 상태와 서버 상태를 구분했다.
- [ ] source of truth가 어디인지 명확하다.
- [ ] 임시 상태와 장기 상태를 구분했다.
- [ ] localStorage나 캐시를 어디까지 허용할지 정리했다.

## 7.2 API 설계
- [ ] 1차 MVP에 필요한 API 후보가 정리되어 있다.
- [ ] 각 API의 목적이 설명되어 있다.
- [ ] 입력/출력/보안 포인트가 최소한 정리되어 있다.
- [ ] API 설계가 과도하게 많지 않고 단계적으로 나뉘어 있다.

---

# 8. 범위 및 단계 설계 체크리스트

## 8.1 범위 제어
- [ ] 설계가 한 번에 너무 많은 기능을 포함하지 않는다.
- [ ] 1차 MVP 범위가 분명하다.
- [ ] 제외 범위가 명시되어 있다.
- [ ] 후속 Phase로 넘길 항목이 정리되어 있다.

## 8.2 단계별 계획
- [ ] Phase 또는 Step 단위로 나뉘어 있다.
- [ ] 각 단계별 목표가 설명되어 있다.
- [ ] 각 단계별 완료 조건이 있다.
- [ ] 각 단계별 예상 변경 범위가 있다.
- [ ] 각 단계가 이전 단계 결과에 자연스럽게 이어진다.

# 9. 리스크 체크리스트

## 9.1 구조 리스크
- [ ] 구조적 모호성이 남아 있는 부분을 적었다.
- [ ] 다음 단계에서 재검토할 구조 포인트를 적었다.
- [ ] 장기적으로 부채가 될 수 있는 임시 구조를 인지하고 있다.

## 9.2 회귀 위험
- [ ] 기존 기능 회귀 가능성을 점검했다.
- [ ] 공통 컴포넌트/공통 레이아웃 영향 가능성을 봤다.
- [ ] 작은 변경처럼 보이지만 영향 범위가 큰 부분을 의식하고 있다.

## 9.3 보안 리스크
- [ ] 민감정보 노출 가능성을 점검했다.
- [ ] 클라이언트/서버 경계 붕괴 가능성을 점검했다.
- [ ] 로그/응답에 과도한 정보가 실릴 가능성을 점검했다.

## 9.4 운영 리스크
- [ ] 운영 공백 가능성을 점검했다.
- [ ] legacy를 너무 빨리 제거하지 않는지 검토했다.
- [ ] 읽기 중심 운영 페이지로 충분한지 검토했다.

# 13. 이 문서의 역할

이 문서는 설계 문서 자체가 아니다.

이 문서는
- 설계가 충분히 구체적인지 보고
- 구현 전에 범위를 제어하고
- 리스크와 누락을 드러내고
- 바로 구현 가능한 수준인지 판단하는
설계 리뷰용 안전장치다.

즉,
마스터 플랜이 방향과 구조를 정리하는 문서라면,
이 체크리스트는 그 설계가 실행 가능한지 검토하는 도구다.

---

# 14. 결론

`office-unify` 프로젝트에서 좋은 설계는
단순히 멋있는 구조도가 아니다.

좋은 설계는 아래를 함께 만족해야 한다.

- 범위가 적절하다
- 프로젝트 원칙과 맞다
- 기존 기능 보호가 반영된다
- 서버/클라이언트/보안 경계가 명확하다
- 단계별 구현이 가능하다
- 검증과 문서화 계획이 있다
- 남은 리스크가 숨겨져 있지 않다

이 체크리스트는 그 기준을 구현 직전 시점에서 확인하기 위한 도구다.
