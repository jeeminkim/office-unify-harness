# Office Unify Implementation Checklist

## 문서 목적
이 문서는 `office-unify` 통합 프로젝트에서
실제 구현 전과 구현 후에 확인해야 할 핵심 항목을 점검하기 위한 체크리스트다.

이 체크리스트의 목적은 다음과 같다.

- 구현 전에 범위와 전제 조건 확인
- 구현 중 하네스 원칙 이탈 방지
- 구현 후 검증/문서화 누락 방지
- 기존 기능 회귀 위험 조기 식별
- Done / Partial Done 판단 보조

이 문서는 분석 문서나 설계 문서를 대체하지 않는다.
이 문서는 실제 구현 실행 직전과 직후에 빠르게 점검하는 실무용 체크리스트다.

---

# 1. 사용 원칙

이 체크리스트는 아래 상황에서 사용한다.

- 1차 구현 시작 전
- 작은 단위 기능 이관 시작 전
- 서버 구조 추가 전
- 새 라우트 또는 메뉴 추가 전
- 구현 직후 검증 전
- 결과 보고 정리 전

원칙은 아래와 같다.

1. 체크리스트는 형식적으로만 체크하지 않는다.
2. 확실하지 않은 항목은 체크하지 말고 메모를 남긴다.
3. “검토 예정”과 “확인 완료”를 구분한다.
4. 체크리스트 결과는 필요하면 보고 문서에 반영한다.
5. 리스크가 있으면 `risk-log`와 함께 본다.

---

# 2. 구현 시작 전 체크리스트

## 2.1 현재 단계 확인
- [ ] 지금 단계가 분석이 아니라 구현 단계임이 분명하다.
- [ ] 분석 결과 또는 설계 문서가 존재한다.
- [ ] 이번 구현 범위가 한 번에 너무 크지 않다.
- [ ] 이번 작업이 전체 통합이 아니라 작은 단위 구현임이 분명하다.
- [ ] 구현 목표가 한 문장으로 설명 가능하다.

## 2.2 범위 확인
- [ ] 이번 턴의 직접 수정 대상이 명확하다.
- [ ] 참조만 할 대상과 실제 수정 대상을 구분했다.
- [ ] 이번 단계에서 하지 않을 항목을 정리했다.
- [ ] 기존 기능을 교체하는 작업이 아니라 추가/분리 중심 작업임을 확인했다.
- [ ] 롤백이 어렵거나 너무 큰 변경은 제외했다.

## 2.3 프로젝트 원칙 확인
- [ ] 최종 제품 본체는 `dev_support`라는 점을 유지하고 있다.
- [ ] `ai_office`는 참조 원본으로 다루고 있다.
- [ ] 기존 `dev_support` 핵심 기능 보호가 우선이다.
- [ ] Discord 결합 로직을 직접 이식하지 않기로 했다.
- [ ] 민감정보는 서버 전용이라는 원칙을 유지하고 있다.
- [ ] 점진적 통합 원칙을 유지하고 있다.

---

# 3. 아키텍처/구조 체크리스트

## 3.1 구조 적합성
- [ ] 이번 변경이 `dev_support` 중심 구조와 충돌하지 않는다.
- [ ] 이번 변경이 다음 단계 확장을 위한 발판이 된다.
- [ ] 기존 구조를 불필요하게 전면 교체하지 않는다.
- [ ] 임시 코드가 장기 구조를 망치지 않는 수준이다.

## 3.2 클라이언트/서버 경계
- [ ] 화면 렌더링/입력/표시는 클라이언트에 둔다.
- [ ] 민감정보/외부 API/판단 로직은 서버에 둔다.
- [ ] 서버 전용 로직이 클라이언트 코드에 섞이지 않는다.
- [ ] API route, service, repository 역할이 과도하게 섞이지 않는다.

## 3.3 Discord 치환 원칙
- [ ] Discord interaction 로직을 그대로 복사하지 않는다.
- [ ] Discord 버튼/모달/ephemeral 개념은 웹 구조로 재해석한다.
- [ ] customId 기반 흐름을 그대로 들여오지 않는다.
- [ ] 필요 시 legacy 또는 adapter 후보로 남긴다.

---

# 4. 기능 이관 체크리스트

## 4.1 순수 로직
- [ ] 이관 대상 중 순수 로직을 먼저 식별했다.
- [ ] 순수 로직은 UI/Discord 종속성 없이 추출 가능한지 확인했다.
- [ ] 추출 가능한 로직과 재설계가 필요한 로직을 구분했다.

## 4.2 서버 전용 로직
- [ ] 외부 API 호출은 서버 전용으로 다룬다.
- [ ] Supabase 접근은 서버 기준으로 본다.
- [ ] provider/service/repository 분리가 필요한지 판단했다.
- [ ] source of truth가 클라이언트로 밀리지 않도록 했다.

## 4.3 운영 기능
- [ ] 운영 기능은 읽기 중심 웹 전환부터 검토했다.
- [ ] 기존 운영 흐름을 한 번에 제거하지 않는다.
- [ ] 운영 공백이 생길 가능성을 점검했다.

---

# 5. 구현 중 체크리스트

## 5.1 변경 단위
- [ ] 한 번에 하나의 논리 단위만 수정하고 있다.
- [ ] UI와 서버, DB와 provider를 한 번에 과도하게 섞지 않는다.
- [ ] 현재 작업 범위 바깥의 리팩토링을 억제하고 있다.
- [ ] 변경 이유를 설명할 수 있다.

## 5.2 기존 기능 보호
- [ ] 기존 업무 도우미 주요 기능에 직접 영향이 없는지 확인했다.
- [ ] 공통 컴포넌트 수정 시 영향 범위를 점검했다.
- [ ] 라우트/레이아웃 변경 시 기존 진입 흐름을 점검했다.
- [ ] 회귀 위험이 보이면 즉시 범위를 줄일 수 있다.

## 5.3 보안/민감정보
- [ ] API key 또는 service role key를 클라이언트에 두지 않는다.
- [ ] 응답 데이터에 과도한 정보가 포함되지 않는다.
- [ ] 로그에 민감정보를 남기지 않는다.
- [ ] 디버깅 편의를 이유로 보안 경계를 무너뜨리지 않는다.

---

# 6. 구현 후 검증 체크리스트

## 6.1 기본 검증
- [ ] lint 수행 가능 여부를 확인했다.
- [ ] build 수행 가능 여부를 확인했다.
- [ ] typecheck 수행 가능 여부를 확인했다.
- [ ] import/경로 오류 가능성을 점검했다.
- [ ] 새 라우트 또는 새 파일 참조 오류를 점검했다.

## 6.2 기능 검증
- [ ] 기존 `dev_support` 핵심 기능 영향 여부를 확인했다.
- [ ] 신규 메뉴 또는 라우트 진입 가능 여부를 확인했다.
- [ ] 신규 API가 있다면 최소 응답 흐름을 점검했다.
- [ ] 서버 구조 추가 시 최소 실행 경로를 점검했다.
- [ ] 에러 상태/빈 상태 처리 누락 여부를 점검했다.

## 6.3 보안/경계 검증
- [ ] 클라이언트/서버 경계가 무너지지 않았는지 확인했다.
- [ ] 민감정보 노출 가능성을 다시 확인했다.
- [ ] Discord 결합 로직을 그대로 가져오지 않았는지 확인했다.
- [ ] localStorage 의존이 과도하지 않은지 점검했다.

---

# 7. 문서화 체크리스트

## 7.1 README
- [ ] README 갱신이 필요한지 판단했다.
- [ ] 구조 변화가 있다면 README 반영 후보를 정리했다.

## 7.2 docs
- [ ] 아키텍처 문서 갱신이 필요한지 판단했다.
- [ ] 워크플로 문서 갱신이 필요한지 판단했다.
- [ ] 계획 문서와 실제 구현 상태 차이를 점검했다.
- [ ] 새 API, 새 라우트, 새 폴더 구조가 생겼다면 문서 반영 후보를 적었다.

## 7.3 보고 문서
- [ ] 설계 요약을 정리할 수 있다.
- [ ] 변경 파일 목록을 정리할 수 있다.
- [ ] 검증한 것과 못한 것을 구분할 수 있다.
- [ ] 남은 리스크와 다음 단계를 적을 수 있다.

# 8. Done / Partial Done 판정 체크리스트

## 8.1 Done에 가까운 상태
- [ ] 이번 구현 목표가 충족되었다.
- [ ] 기본 검증을 수행했다.
- [ ] 문서 반영 필요 여부를 정리했다.
- [ ] 남은 리스크를 공개했다.
- [ ] 결과 보고 형식으로 정리 가능하다.

## 8.2 Partial Done에 해당하는 상태
- [ ] 구조 골격만 반영했다.
- [ ] 구현은 했지만 검증이 부족하다.
- [ ] 환경 의존으로 일부 확인하지 못했다.
- [ ] 후속 작업 없이는 완료라고 보기 어렵다.
- [ ] 문서 반영이 아직 남아 있다.

원칙:
- 위 항목 중 Partial Done 성격이 크면 Done처럼 표현하지 않는다.

- [ ] 완료보다 Partial Done 가능성도 열어두고 있는가

# 12. 이 문서의 역할

이 문서는 구현을 대신하는 문서가 아니다.

이 문서는
- 구현 전에 범위를 제어하고
- 구현 중 원칙 이탈을 막고
- 구현 후 검증/문서화 누락을 줄이는
실행 체크 도구다.

즉,
설계 문서가 방향을 잡는 문서라면,
이 체크리스트는 실제 실행 순간의 안전장치다.

---

# 13. 결론

`office-unify` 프로젝트에서 구현은
“코드를 바꾸는 행위”만으로 끝나지 않는다.

좋은 구현은 아래를 함께 만족해야 한다.

- 범위가 작다
- 구조 원칙을 지킨다
- 기존 기능을 보호한다
- 검증 가능하다
- 문서화와 리스크 공개가 뒤따른다

이 체크리스트는 그 기준을 실제 구현 순간에 빠르게 점검하기 위한 도구다.
