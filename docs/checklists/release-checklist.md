# Office Unify Release Checklist

## 문서 목적
이 문서는 `office-unify` 통합 프로젝트에서
배포 또는 공개 가능한 상태인지 점검하기 위한 릴리즈 체크리스트다.

이 체크리스트의 목적은 다음과 같다.

- 배포 직전 품질 기준 확인
- 기존 기능 회귀 여부 재점검
- 보안/환경변수/운영 리스크 점검
- 문서 반영 누락 방지
- Done과 Release Ready를 구분하기 위한 기준 마련

이 문서는 단순 구현 완료 여부를 보는 문서가 아니다.
이 문서는 “지금 상태를 외부 사용 또는 다음 운영 단계로 넘겨도 되는가”를 판단하는 실무용 체크리스트다.

---

# 1. 사용 시점

이 체크리스트는 아래 상황에서 사용한다.

- 1차 MVP 통합 결과를 릴리즈 후보로 볼 때
- 주요 기능 이관 후 내부 공개 직전
- 새 메뉴/라우트/API를 포함한 변경을 반영한 뒤
- Vercel 또는 기타 배포 환경 반영 전
- 사용자 또는 운영자가 실제로 접근할 수 있는 상태로 열기 전

원칙은 아래와 같다.

1. Done과 Release Ready는 동일하지 않다.
2. 구현이 끝났더라도 운영/보안/문서 조건이 부족하면 릴리즈하지 않는다.
3. 릴리즈 판단이 애매하면 Partial Release 또는 내부 검증용 상태로 본다.
4. 릴리즈 전에는 회귀 위험, 보안 위험, 운영 공백 위험을 다시 본다.

---

# 2. 릴리즈 기본 적합성 체크리스트

## 2.1 현재 릴리즈 범위 확인
- [ ] 이번 릴리즈의 범위를 한 문장으로 설명할 수 있다.
- [ ] 이번 릴리즈에 포함되는 기능이 명확하다.
- [ ] 이번 릴리즈에 포함되지 않는 기능도 명확하다.
- [ ] 이번 릴리즈가 전체 통합 완성이 아니라 특정 단계 공개라는 점이 정리되어 있다.
- [ ] 사용자/운영자에게 어떤 변화가 생기는지 설명 가능하다.

## 2.2 릴리즈 목적 확인
- [ ] 왜 지금 이 상태를 릴리즈하려는지 설명 가능하다.
- [ ] 내부 검증용 릴리즈인지, 실제 사용 가능한 릴리즈인지 구분되어 있다.
- [ ] 이번 릴리즈가 다음 단계의 발판 역할을 하는지 설명 가능하다.
- [ ] 릴리즈 범위가 과도하게 크지 않다.

---

# 3. 기능 체크리스트

## 3.1 기존 기능 보호
- [ ] 기존 `dev_support` 핵심 기능이 유지된다.
- [ ] flow/sql/ts 생성 등 기존 주요 기능의 회귀 가능성을 점검했다.
- [ ] 기존 주요 진입 경로가 깨지지 않았다.
- [ ] 공통 레이아웃/공통 컴포넌트 변경이 기존 기능을 해치지 않는다.

## 3.2 신규 기능 점검
- [ ] 신규 메뉴 또는 진입점이 의도대로 노출된다.
- [ ] 신규 라우트가 진입 가능하다.
- [ ] 신규 화면이 최소한 깨지지 않고 렌더링된다.
- [ ] 신규 기능이 “준비 중/placeholder” 상태라면 그 상태가 사용자에게 명확하다.
- [ ] 미완성 기능이 실제 완성 기능처럼 보이지 않게 처리했다.

## 3.3 통합 방향 정합성
- [ ] 이번 릴리즈가 `dev_support` 중심 통합 원칙과 맞다.
- [ ] `ai_office` 로직을 무분별하게 직접 이식하지 않았다.
- [ ] Discord 결합 로직은 직접 노출되지 않는다.
- [ ] 이번 릴리즈가 다음 통합 단계를 위한 구조적 발판이 된다.

---

# 4. 아키텍처/경계 체크리스트

## 4.1 클라이언트/서버 경계
- [ ] 클라이언트는 화면/입력/표시 중심 역할을 유지한다.
- [ ] 서버는 민감정보, 외부 연동, 핵심 처리 로직을 담당한다.
- [ ] 서버 전용 로직이 클라이언트 코드에 섞이지 않았다.
- [ ] API, service, repository, provider 역할이 과도하게 섞이지 않았다.

## 4.2 Discord 치환 상태
- [ ] Discord button/modal/interaction 구조를 그대로 노출하지 않는다.
- [ ] Discord 개념은 웹 UI/웹 액션으로 재해석되었다.
- [ ] customId 기반 흐름을 그대로 끌고 오지 않았다.
- [ ] legacy 또는 adapter가 필요한 경우 그 상태를 문서로 남겼다.

## 4.3 상태 관리
- [ ] 장기 데이터의 source of truth가 서버 기준으로 유지된다.
- [ ] localStorage 의존이 과도하지 않다.
- [ ] 클라이언트 상태와 서버 상태가 충돌하지 않도록 설계되어 있다.
- [ ] 임시 상태와 실제 운영 데이터가 혼동되지 않는다.

---

# 5. 보안 체크리스트

## 5.1 민감정보
- [ ] API Key가 클라이언트에 노출되지 않는다.
- [ ] Supabase service role key가 클라이언트에 포함되지 않는다.
- [ ] 환경변수 사용 위치가 적절하다.
- [ ] 디버그 코드나 임시 로그에 민감정보가 남아 있지 않다.

## 5.2 응답/데이터 노출
- [ ] API 응답에 불필요한 내부 정보가 포함되지 않는다.
- [ ] 사용자에게 필요한 최소 데이터만 노출한다.
- [ ] 운영/관리 목적 데이터가 일반 사용자에게 노출되지 않는다.
- [ ] 에러 메시지가 내부 구조를 과도하게 드러내지 않는다.

## 5.3 운영 보안
- [ ] 운영용 페이지 또는 상태 페이지가 있다면 접근 범위를 고려했다.
- [ ] 인증/권한이 아직 미구현이라면 최소한 공개 범위를 인지하고 있다.
- [ ] 향후 인증/권한 구조가 필요하면 리스크로 남겨 두었다.

---

# 6. 검증 체크리스트

## 6.1 기본 검증
- [ ] lint 수행 여부를 확인했다.
- [ ] build 수행 여부를 확인했다.
- [ ] typecheck 수행 여부를 확인했다.
- [ ] import/경로 정합성을 점검했다.
- [ ] 새 라우트 및 새 파일 참조 오류 여부를 점검했다.

## 6.2 기능 검증
- [ ] 기존 업무 도우미 핵심 흐름을 최소한 확인했다.
- [ ] 신규 메뉴/라우트 진입을 확인했다.
- [ ] 신규 API가 있다면 최소 응답 구조를 확인했다.
- [ ] 빈 상태/에러 상태 처리를 최소한 점검했다.
- [ ] 통합 UI에서 기존과 신규 기능이 충돌하지 않는지 점검했다.

## 6.3 환경 의존 검증
- [ ] 로컬에서만 확인한 항목과 배포 환경 검증 항목을 구분했다.
- [ ] 실제 외부 API/Supabase 연동 여부를 별도로 구분했다.
- [ ] 환경 미비로 확인하지 못한 항목을 정리했다.
- [ ] 미검증 항목이 릴리즈 허용 범위를 넘어서는지 판단했다.

---

# 7. 문서화 체크리스트

## 7.1 README
- [ ] README 갱신 필요 여부를 판단했다.
- [ ] 구조/실행/기능 설명 변화가 있다면 반영했다.
- [ ] 새 진입점 또는 새 작업 방식이 있다면 반영했다.

## 7.2 docs
- [ ] 아키텍처 문서 반영 필요 여부를 판단했다.
- [ ] 워크플로/체크리스트/계획 문서의 상태를 점검했다.
- [ ] 마이그레이션 단계 진척이 있다면 관련 문서를 갱신했다.
- [ ] 새 API/폴더 구조/라우트가 생겼다면 반영 후보를 적었다.

## 7.3 릴리즈 설명
- [ ] 이번 릴리즈의 포함/제외 범위를 설명할 수 있다.
- [ ] 사용자 또는 운영자 관점의 변화 요약을 적을 수 있다.
- [ ] 아직 미완성인 부분을 명확히 안내할 수 있다.

# 8. 운영/릴리즈 준비 체크리스트

## 8.1 운영 영향
- [ ] 이번 릴리즈가 기존 운영 흐름을 막지 않는다.
- [ ] Discord 운영 기능을 너무 빨리 제거하지 않았다.
- [ ] 운영 공백 가능성을 검토했다.
- [ ] 읽기 중심 운영 페이지가 있다면 최소 목적을 충족한다.

## 8.2 배포 준비
- [ ] 필요한 환경변수 목록이 정리되어 있다.
- [ ] 배포 환경에서 필요한 설정 변경이 있는지 파악했다.
- [ ] 빌드/배포 명령 기준이 정리되어 있다.
- [ ] 배포 후 최소 점검 항목이 정리되어 있다.

## 8.3 롤백 가능성
- [ ] 문제가 생겼을 때 롤백 또는 비활성화가 가능한 범위다.
- [ ] 릴리즈 범위가 너무 커서 롤백이 어려운 상태가 아니다.
- [ ] 위험이 크면 기능 노출 범위를 줄이는 대안을 고려했다.

# 9. 릴리즈 판정 체크리스트

## 9.1 Release Ready에 가까운 상태
- [ ] 릴리즈 범위가 명확하다.
- [ ] 기존 기능 회귀 가능성이 허용 범위 내다.
- [ ] 기본 검증을 수행했다.
- [ ] 민감정보/보안 점검을 했다.
- [ ] 문서 반영 필요 여부를 정리했다.
- [ ] 남은 리스크를 공개했다.
- [ ] 사용자/운영자에게 어떤 상태인지 설명 가능하다.

## 9.2 Partial Release에 해당하는 상태
- [ ] 구조 골격은 반영됐지만 기능 확인이 제한적이다.
- [ ] UI 진입은 가능하지만 실제 데이터 연동은 미확인이다.
- [ ] 내부 검증용 공개 수준이다.
- [ ] 문서 또는 운영 준비가 덜 끝났다.
- [ ] 후속 검증 없이는 정식 릴리즈로 보기 어렵다.

## 9.3 Release Blocker 예시
- [ ] 기존 핵심 기능 회귀 가능성이 높다.
- [ ] build/typecheck 등 기본 검증이 실패한다.
- [ ] 민감정보 노출 가능성이 있다.
- [ ] 신규 기능이 실제로 오해를 일으킬 정도로 불안정하다.
- [ ] 운영 공백 가능성이 크다.
- [ ] 문서와 실제 구조가 심하게 어긋난다.

원칙:
- Release Blocker가 남아 있으면 릴리즈하지 않는다.

- [ ] 신규 메뉴/라우트는 최소 정상 진입 가능한가
- [ ] Discord 직접 이식 흔적이 없는가
- [ ] 민감정보 노출 위험이 없는가
- [ ] 클라이언트/서버 경계가 유지되는가
- [ ] 기본 검증 결과를 설명할 수 있는가
- [ ] 미검증 항목을 숨기지 않았는가
- [ ] README/docs 반영 필요 여부를 판단했는가
- [ ] 남은 리스크를 공개했는가
- [ ] 롤백 또는 범위 축소가 가능한가
- [ ] Release Ready인지 Partial Release인지 구분했는가

# 12. 이 문서의 역할

이 문서는 단순 QA 체크리스트가 아니다.

이 문서는
- 구현 완료와 릴리즈 가능 상태를 구분하고
- 보안/운영/회귀/문서 조건을 함께 보고
- 실제 공개 또는 배포 판단을 더 신중하게 하기 위한
릴리즈 안전장치다.

즉,
Done이 곧바로 Release Ready를 의미하지 않는다는 점을
실무적으로 확인하는 도구다.

# 13. 결론

`office-unify` 프로젝트에서 릴리즈는
코드를 합쳤다는 이유만으로 진행하면 안 된다.

좋은 릴리즈는 아래를 함께 만족해야 한다.

- 범위가 명확하다
- 기존 기능이 보호된다
- 신규 기능 상태가 솔직하게 표현된다
- 보안 경계가 유지된다
- 검증과 문서가 뒤따른다
- 남은 리스크가 공개된다

이 체크리스트는 그 기준을 릴리즈 직전에 빠르게 점검하기 위한 도구다.
