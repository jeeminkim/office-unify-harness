# Office Unify Terminology

## 문서 목적
이 문서는 `office-unify-harness` 및 향후 실제 통합 프로젝트에서 사용하는 핵심 용어를 정의한다.

목적은 다음과 같다.

- 문서 간 표현 흔들림 방지
- Cursor 프롬프트와 계획 문서의 의미 통일
- 설계/구현/검증 시 해석 차이 최소화
- GPT Builder 확장 시 상위 지식 기준 마련

이 문서에서 정의한 용어는
프로젝트 헌장, 아키텍처 문서, 마이그레이션 계획, 프롬프트 템플릿 전반에 공통 적용한다.

---

# 1. 제품/저장소 관련 용어

## 1.1 Office Unify
`ai_office`와 `dev_support`를 별도 서비스가 아닌
하나의 통합 웹 서비스로 재구성하려는 전체 프로젝트 또는 그 최종 제품 방향을 뜻한다.

문맥에 따라 아래 두 의미로 쓰일 수 있다.

- 통합 프로젝트 전체
- 통합 이후의 최종 웹 앱 제품 방향

혼동이 생기면 아래처럼 구분한다.

- `Office Unify 프로젝트`
- `Office Unify 제품`

---

## 1.2 dev_support
현재 React/Next.js 기반의 업무 도우미 앱 저장소를 뜻한다.

이 프로젝트에서는 다음 의미를 가진다.

- 최종 제품의 구조적 중심축
- 통합 이후 제품 본체
- UI/웹 구조의 기준점
- 실제 통합 작업의 우선 수정 대상

원칙상 `dev_support`는 “최종 제품 본체”로 본다.

---

## 1.3 ai_office
기존 Discord 기반 AI Office 서비스 저장소를 뜻한다.

이 프로젝트에서는 다음 의미를 가진다.

- 기능 원천
- 참조 소스
- 분석 대상
- 서버 로직 흡수 후보의 원천

원칙상 `ai_office`는 “기능 원천/참조 원본”으로 보며,
초기 통합 단계에서 직접 수정 대상이 되지 않는 것이 기본이다.

---

## 1.4 통합 제품 본체
최종적으로 하나의 제품으로 운영될 때
구조적 중심이 되는 앱 또는 저장소를 의미한다.

현재 프로젝트에서는 `dev_support`가 이에 해당한다.

즉, 기능 일부가 `ai_office`에서 오더라도
제품 본체는 `dev_support` 기준으로 설계한다.

---

## 1.5 참조 원본
기능, 로직, 흐름, 데이터 처리 방식 등을 분석하고 재사용하기 위해 참고하는 원본 저장소 또는 코드 집합을 뜻한다.

현재 프로젝트에서는 `ai_office`가 대표적인 참조 원본이다.

참조 원본은 “그대로 복사할 대상”이 아니라,
분해/분류/추출/재설계의 대상이다.

---

# 2. 아키텍처 관련 용어

## 2.1 순수 로직
특정 UI 프레임워크, Discord, 페이지 구조, 네트워크 진입점, 런타임 환경에 강하게 묶이지 않은
재사용 가능한 핵심 로직을 뜻한다.

예시:
- 데이터 가공
- 판단 규칙
- 점수 계산
- 결과 조합
- provider 선택 로직
- 도메인별 처리 규칙

순수 로직은 우선적인 재사용 후보로 본다.

---

## 2.2 Discord 결합 로직
Discord 인터랙션 모델이나 Discord 메시징 방식에 직접 의존하는 코드를 뜻한다.

예시:
- interaction 처리
- customId 라우팅
- ephemeral reply
- panel restore
- Discord 버튼/모달 흐름
- Discord 메시지 포맷

이 범주의 로직은 원칙적으로 웹에 직접 이식하지 않는다.

---

## 2.3 웹 치환
기존 Discord 결합 로직이나 채널 의존 흐름을
웹 앱에 맞는 구조와 경험으로 재설계하는 것을 뜻한다.

예시:
- Discord button → 웹 버튼
- modal → 웹 폼/다이얼로그
- ephemeral reply → 사용자 전용 결과 패널
- interaction routing → 웹 라우트/API 핸들러

웹 치환은 “기술적 복제”가 아니라 “기능 목적의 재구성”이다.

---

## 2.4 서버 계층
민감정보, 외부 연동, 데이터 처리, 핵심 판단 로직 등을 담당하는
웹 앱의 서버 전용 구조를 뜻한다.

예시:
- services
- repositories
- providers
- contracts
- server utilities
- API handlers

이 프로젝트에서는 `ai_office`의 재사용 가능한 핵심 기능을
가능한 한 서버 계층으로 흡수하는 방향을 기본으로 삼는다.

---

## 2.5 클라이언트 계층
사용자에게 보이는 웹 화면, 입력 수집, 결과 시각화, 상태 표시 등을 담당하는 계층을 뜻한다.

예시:
- 페이지
- 컴포넌트
- 폼 UI
- 상태 뷰
- 결과 패널
- 로딩/에러 UI

클라이언트 계층은 민감정보를 직접 다루지 않는다.

---

## 2.6 Provider
LLM, 외부 API, 또는 특정 외부 서비스와의 연결을 담당하는 추상화 계층을 뜻한다.

예시:
- OpenAI provider
- Gemini provider
- 외부 시세 조회 provider

Provider는 외부 종속성을 한 곳에 모아두기 위한 구조적 장치다.

---

## 2.7 Repository
DB 또는 영속 저장소와의 접근을 담당하는 계층을 뜻한다.

예시:
- Supabase repository
- portfolio repository
- decision history repository

Repository는 저장소 접근을 분리하여
서비스 로직과 영속성 로직이 섞이지 않게 만든다.

---

## 2.8 Service
핵심 비즈니스 규칙이나 기능 실행 흐름을 담당하는 계층을 뜻한다.

예시:
- 포트폴리오 조회 서비스
- AI 토론 실행 서비스
- 트렌드 분석 서비스

Service는 UI, DB, Provider 사이를 연결하되
도메인 관점의 중심 로직을 담는 계층이다.

---

## 2.9 Contract
계층 간 데이터 형태와 인터페이스를 명시하는 구조를 뜻한다.

예시:
- request/response 타입
- DTO
- domain interface
- provider interface

Contract는 시스템 내부의 연결 규칙을 명확히 하기 위한 장치다.

---

## 2.10 Source of Truth
특정 데이터의 최종 기준이 되는 원천 저장 위치 또는 권한 있는 상태를 뜻한다.

이 프로젝트에서는
투자/운영/이력 등 장기 데이터는 서버 측 source of truth를 우선한다.

localStorage나 일시적 클라이언트 상태는 보조 수단이지,
최종 기준이 아니다.

---

# 3. 마이그레이션 관련 용어

## 3.1 마이그레이션
기존 `ai_office`의 기능/로직/구조 일부를
`dev_support` 중심의 웹 구조로 옮기고 재구성하는 과정을 뜻한다.

여기서 마이그레이션은
단순 파일 복사가 아니라 아래를 포함한다.

- 분석
- 분류
- 추출
- 재설계
- 검증
- 문서화

---

## 3.2 점진적 통합
한 번에 모든 기능을 옮기지 않고,
작은 단위로 나누어 순차적으로 흡수하는 전략을 뜻한다.

이 프로젝트의 기본 방식은 점진적 통합이다.

---

## 3.3 1차 MVP 통합
현재 단계에서 우선적으로 웹 전환 가치가 높은 기능만
최소 범위로 통합하는 초기 목표를 뜻한다.

현재 기준 예시는 아래와 같다.

- 기존 `dev_support` 기능 유지
- AI Office 메뉴/레이아웃 추가
- 포트폴리오 조회
- AI 토론 실행
- 트렌드 분석 실행
- 운영 상태 읽기 중심 페이지

---

## 3.4 Legacy
당장 제거하지 않고 일정 기간 유지하는 기존 구조 또는 기능을 뜻한다.

예시:
- Discord 전용 운영 흐름
- 기존 Discord 관리자 액션
- 전면 개편 전 임시 유지 코드

Legacy는 “문제 코드”라는 뜻이 아니라,
이행 과정에서 당장 없애지 않는 구조라는 의미다.

---

## 3.5 Legacy Adapter
기존 시스템과 새 시스템을 일정 기간 연결하기 위한 완충 계층을 뜻한다.

예시:
- Discord 흐름을 완전히 제거하지 않고
  신규 웹 구조와 병행 가능한 형태로 남기는 연결 장치

이 프로젝트에서 Legacy Adapter는
급격한 전면 교체를 피하기 위한 안전장치다.

---

## 3.6 이관 후보
기존 저장소에서 재사용 또는 재설계를 통해
통합 제품으로 옮길 가치가 있는 로직, 서비스, 유틸, 흐름을 뜻한다.

---

## 3.7 직접 이식 금지 대상
구조적으로 현재 제품에 그대로 옮기면
문제를 일으킬 가능성이 높아, 먼저 분리/재설계/격리가 필요한 대상을 뜻한다.

예시:
- Discord 결합 코드
- 민감정보가 섞인 클라이언트 지향 코드
- UI/DB/Provider가 뒤섞인 함수
- 운영 환경에 강하게 종속된 흐름

---

# 4. 하네스 관련 용어

## 4.1 프롬프트 엔지니어링
현재 턴에서 AI에게 무엇을 하라고 지시할지 설계하는 행위를 뜻한다.

예시:
- “지금은 분석만 하고 코드 수정은 하지 마라.”
- “변경 파일 목록과 검증 방법까지 정리하라.”

---

## 4.2 컨텍스트 엔지니어링
프로젝트의 배경, 제약, 목표, 우선순위 등
AI가 지속적으로 알고 있어야 할 맥락을 구조화하는 행위를 뜻한다.

예시:
- `dev_support` 중심 통합
- `ai_office`는 참조 원본
- 기존 기능 보호 최우선
- 민감정보는 서버 전용

---

## 4.3 하네스 엔지니어링
AI가 안정적으로 일하도록
프롬프트, 규칙, 계획, 검증, 결과 보고 형식까지 포함한
작업 전체의 레일을 설계하는 방식을 뜻한다.

이 프로젝트에서 하네스 엔지니어링은 다음을 포함한다.

- `.cursor/rules/`
- `.cursor/plans/`
- 체크리스트
- 검증 기준
- 보고 형식
- 문서화 원칙
- 리스크 공개 기준

---

## 4.4 Project Charter
프로젝트의 최상위 목적과 절대 원칙을 정의한 문서를 뜻한다.

이 저장소에서는 `.cursor/rules/00-project-charter.md`가 이에 해당한다.

---

## 4.5 Architecture Boundaries
어떤 로직이 어디에 있어야 하는지,
무엇을 직접 옮기면 안 되는지,
보안/구조 경계를 어떻게 지켜야 하는지를 정의한 기준을 뜻한다.

이 저장소에서는 `.cursor/rules/10-architecture-boundaries.md`가 이에 해당한다.

---

## 4.6 Migration Workflow
분석 → 설계 → 구현 → 검증 → 문서화 순서와
변경 단위를 어떻게 제어할지 정한 실행 규칙을 뜻한다.

이 저장소에서는 `.cursor/rules/20-migration-workflow.md`가 이에 해당한다.

---

## 4.7 Definition of Done
어떤 상태를 완료라고 볼 수 있는지 정한 기준을 뜻한다.

이 저장소에서는 `.cursor/rules/30-validation-and-done.md`가 이에 해당한다.

---

## 4.8 Master Plan
실제 통합 작업 전에
범위, 구조, 리스크, 구현 단계, 검증 계획을 정리하는 계획 문서를 뜻한다.

이 저장소에서는 `.cursor/plans/office-unify-master-plan.template.md`가 이에 해당한다.

---

# 5. 운영/품질 관련 용어

## 5.1 완료
작업 목적 달성, 검증 수행, 문서 반영, 리스크 공개가 모두 충족된 상태를 뜻한다.

단순 구현 반영만으로는 완료가 아니다.

---

## 5.2 부분 완료
구현은 되었지만 검증이 부족하거나,
설계만 끝났거나,
후속 작업이 필수인 상태를 뜻한다.

부분 완료는 완료와 구분해서 표현해야 한다.

---

## 5.3 검증
변경 사항이 실제로 문제없이 동작하는지 확인하기 위한 활동을 뜻한다.

예시:
- lint
- build
- typecheck
- smoke test
- 주요 화면 확인
- API 응답 확인

---

## 5.4 리스크
현재 구조나 변경으로 인해 향후 문제가 생길 가능성이 있는 요소를 뜻한다.

리스크는 숨기지 않고 문서/보고에 포함해야 한다.

---

## 5.5 회귀 위험
새로운 변경 때문에 기존 기능이 깨질 가능성을 뜻한다.

이 프로젝트에서는 회귀 위험을 낮추는 것이 매우 중요하다.

---

# 7. 결론

이 문서의 목적은 단순한 단어 사전이 아니다.

이 문서는
- 프로젝트의 해석 차이를 줄이고
- AI 도구의 판단 흔들림을 줄이며
- 설계/구현/문서화 전반의 일관성을 높이기 위한
공통 용어 기준서다.

앞으로 `office-unify` 관련 문서와 작업은
가능한 한 이 용어 정의를 기준으로 표현을 맞춘다.
