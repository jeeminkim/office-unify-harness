# Office Unify Context Engineering Guide

## 문서 목적
이 문서는 `office-unify` 프로젝트에서
컨텍스트 엔지니어링을 어떻게 이해하고 운영할지 정리하는 실무 가이드다.

이 프로젝트에서 컨텍스트는 단순 참고사항이 아니다.
컨텍스트는 AI가 매번 같은 방향으로 판단하게 만드는
상시 배경이자 구조적 제약이다.

따라서 이 문서는 아래를 설명한다.

- 이 프로젝트에서 컨텍스트 엔지니어링이 무엇인지
- 왜 프롬프트만으로는 부족한지
- 어떤 정보가 상시 컨텍스트여야 하는지
- 컨텍스트를 어떤 문서와 구조로 관리해야 하는지
- Cursor와 GPT Builder에서 각각 어떻게 활용해야 하는지

---

# 1. 이 프로젝트에서 컨텍스트 엔지니어링이란

이 프로젝트에서 컨텍스트 엔지니어링은
AI가 현재 턴 이전에도 알고 있어야 하는 배경, 원칙, 제약, 우선순위를
구조적으로 정리하고 유지하는 행위를 뜻한다.

즉, 컨텍스트는 아래와 같은 것을 포함한다.

- 프로젝트의 최종 목표
- 어떤 저장소가 본체인지
- 어떤 저장소가 참조 원본인지
- 무엇을 바로 옮기면 안 되는지
- 어떤 기능을 먼저 지켜야 하는지
- 클라이언트/서버 경계
- 보안상 지켜야 할 원칙
- 마이그레이션의 기본 순서
- 결과를 어떤 형식으로 정리해야 하는지

컨텍스트는 “이번 턴의 지시”가 아니라,
**매 턴의 판단을 흔들리지 않게 하는 상시 기준**이다.

---

# 2. 왜 컨텍스트 엔지니어링이 필요한가

`office-unify` 프로젝트는 단순한 기능 추가 작업이 아니다.

아래를 동시에 고려해야 한다.

- `ai_office`와 `dev_support`의 역할 차이
- `dev_support` 중심 통합 원칙
- 기존 기능 보호
- Discord 결합 로직 분리
- 웹 구조로의 재설계
- 서버/클라이언트 경계
- 보안과 민감정보 처리
- 점진적 마이그레이션
- 검증과 문서화

이런 정보는 매번 프롬프트에 길게 다시 쓰면 비효율적이고,
누락되거나 해석이 흔들릴 수 있다.

따라서 이 프로젝트는
중요한 배경 정보를 **상시 컨텍스트로 구조화**해야 한다.

즉,
프롬프트가 “이번 턴에 뭘 할까”를 정한다면,
컨텍스트는 “무슨 원칙 아래서 할까”를 정한다.

---

# 3. 프롬프트 엔지니어링과의 차이

이 프로젝트에서 프롬프트 엔지니어링과 컨텍스트 엔지니어링은 분리해서 본다.

## 프롬프트 엔지니어링
현재 턴의 작업 지시를 설계한다.

예:
- 지금은 분석만 하라
- 지금은 설계만 하라
- 지금은 가장 안전한 1차 구현만 하라
- 지금은 검증과 문서화만 하라

## 컨텍스트 엔지니어링
현재 턴과 무관하게 항상 유지해야 하는 배경을 설계한다.

예:
- 최종 제품 본체는 `dev_support`
- `ai_office`는 참조 원본
- Discord 결합 로직은 직접 이식 금지
- 기존 기능 보호 최우선
- 민감정보는 서버 전용
- 점진적 통합 원칙 유지

즉,
프롬프트는 현재 작업의 방향이고,
컨텍스트는 전체 판단의 기준점이다.

---

# 4. 하네스 엔지니어링과의 관계

이 프로젝트에서는 아래 관계로 본다.

## 프롬프트 엔지니어링
현재 턴의 작업 계약

## 컨텍스트 엔지니어링
항상 유지되는 배경과 제약

## 하네스 엔지니어링
프롬프트와 컨텍스트를 포함해
규칙, 계획, 검증, 문서화, 결과 보고까지 묶은
작업 전체의 운영 체계

즉,
- 프롬프트는 움직임
- 컨텍스트는 방향 감각
- 하네스는 전체 레일

컨텍스트 엔지니어링은
하네스 엔지니어링의 핵심 구성 요소 중 하나다.

---

# 5. 이 프로젝트에서 반드시 유지되어야 하는 핵심 컨텍스트

아래는 `office-unify` 프로젝트에서
항상 유지되어야 하는 핵심 컨텍스트다.

## 5.1 제품 중심축
- 최종 제품 본체는 `dev_support`
- `ai_office`는 기능 원천이자 참조 원본
- 구조의 기준은 `dev_support`
- 기능 흡수의 원천 중 일부는 `ai_office`

## 5.2 통합 방향
- `dev_support` 중심 통합
- `ai_office` 서버 로직 흡수
- Discord UI는 웹 구조로 치환
- Discord 결합 로직은 직접 이식 금지

## 5.3 보호 대상
- 기존 `dev_support` 핵심 기능
- 기존 업무 도우미의 안정성
- 민감정보의 서버 전용 유지
- 클라이언트/서버 경계

## 5.4 구조 원칙
- 순수 로직 우선 재사용
- provider / service / repository / contract 분리
- 장기 데이터는 서버 중심 관리
- localStorage는 보조 수단

## 5.5 운영 원칙
- 분석 → 설계 → 구현 → 검증 → 문서화 순서
- 한 번에 작은 단위만 변경
- 검증 없는 완료 선언 금지
- 문서와 리스크 공개 필수

이 다섯 묶음은
이 프로젝트의 핵심 상시 컨텍스트다.

---

# 6. 어떤 정보가 좋은 컨텍스트인가

좋은 컨텍스트는 아래 성격을 가진다.

## 6.1 장기적으로 유효하다
한두 턴만 필요한 정보가 아니라,
여러 단계에서 반복적으로 의미가 있다.

예:
- `dev_support`가 본체라는 점
- Discord 직접 이식 금지
- 기존 기능 보호 우선

## 6.2 판단을 바꾼다
이 정보가 있느냐 없느냐에 따라
설계 또는 구현 선택이 달라진다.

예:
- 민감정보 서버 전용
- source of truth는 서버
- 점진적 통합 우선

## 6.3 문서화 가치가 있다
사람이 봐도 기준점 역할을 한다.

예:
- 용어 정의
- 아키텍처 경계
- 완료 기준
- 작업 순서

## 6.4 반복 설명을 줄인다
매 프롬프트마다 다시 길게 설명할 필요가 줄어든다.

즉,
좋은 컨텍스트는
자주 쓰이고, 판단에 영향을 주고, 오래 유지된다.

---

# 7. 나쁜 컨텍스트의 특징

아래 정보는 상시 컨텍스트로 두기 어렵거나 불필요할 수 있다.

## 7.1 너무 일회적이다
예:
- 이번 턴에만 쓰는 파일 목록
- 임시 비교 결과
- 한 번의 오류 로그 일부

이런 정보는 작업 프롬프트 또는 작업 결과 문서로 두는 편이 낫다.

## 7.2 너무 세부적이다
예:
- 특정 구현의 아주 작은 코드 조각
- 임시 테스트 출력
- 로컬 한 번성 경로 메모

이런 정보는 상시 컨텍스트가 아니라
작업 메모나 구현 산출물에 가깝다.

## 7.3 자주 바뀐다
자주 바뀌는 정보를 상시 컨텍스트로 두면
오히려 혼선을 일으킬 수 있다.

예:
- 임시 브랜치명
- 매 턴 달라지는 변경 파일 목록
- 아직 확정되지 않은 구현 세부사항

즉,
컨텍스트는 “오래 유지되는 기준” 위주여야 한다.

---

# 8. 이 프로젝트에서 컨텍스트를 담는 주요 문서

이 프로젝트에서는 컨텍스트를 말로만 유지하지 않고,
문서와 규칙 파일에 고정한다.

## 핵심 컨텍스트 문서

### `README.md`
- 저장소의 목적
- 하네스 저장소와 실제 작업 루트의 관계
- 전체 운영 방향

### `docs/overview/project-vision.md`
- 왜 통합하는가
- 어떤 제품을 만들고 싶은가
- 장기 방향성

### `docs/overview/terminology.md`
- 용어 기준
- 표현 일관성

### `.cursor/rules/00-project-charter.md`
- 최상위 프로젝트 원칙
- 금지 사항
- 산출물 기준

### `.cursor/rules/10-architecture-boundaries.md`
- 아키텍처 경계
- 서버/클라이언트 분리
- 순수 로직/Discord 결합 로직 구분

### `.cursor/rules/20-migration-workflow.md`
- 작업 순서
- 구현 단위 제어

### `.cursor/rules/30-validation-and-done.md`
- 완료 기준
- 검증과 보고 원칙

즉,
컨텍스트는 문서화되고 고정된 형태로 존재해야 한다.

---

# 9. Cursor에서의 컨텍스트 운영 방식

Cursor에서는 실제 작업 루트의 구조와 규칙 파일이
가장 중요한 컨텍스트 역할을 한다.

## 권장 방식
- 실제 작업 루트에 `.cursor/rules/` 반영
- 실제 작업 루트에 `.cursor/plans/` 반영
- 분석/설계/구현/검증 프롬프트를 분리
- 프롬프트에서는 현재 목표만 짧게 주고,
  상위 원칙은 규칙 파일이 담당하게 함

즉,
Cursor에서는 컨텍스트를
“프로젝트 안에 들어 있는 상시 규칙”으로 다루는 것이 좋다.

---

# 10. GPT Builder에서의 컨텍스트 운영 방식

GPT Builder는 Cursor처럼 실제 코드베이스를 직접 다루는 실행 도구가 아니다.
따라서 GPT Builder에서는
판단 기준과 설명 기준 중심의 컨텍스트가 더 중요하다.

## GPT Builder에 적합한 컨텍스트
- 제품 비전
- 용어 정의
- 최상위 원칙
- 아키텍처 경계
- 작업 순서
- 완료 기준

## GPT Builder에 덜 적합한 것
- 로컬 파일 구조의 세부 구현 메모
- 매 턴 바뀌는 파일 목록
- 임시 작업 로그
- 환경 특화 실행 흔적

즉,
GPT Builder는
코드 실행 문맥보다는
판단과 설명의 상시 컨텍스트 저장소에 가깝다.

---

# 11. 컨텍스트를 관리할 때의 실무 원칙

## 11.1 먼저 문서화하고 반복 사용한다
좋은 기준은 한 번 쓰고 버리지 말고,
문서로 고정하고 재사용한다.

## 11.2 프로젝트 원칙과 구현 메모를 분리한다
원칙은 상시 컨텍스트,
구현 메모는 계획 문서 또는 보고 문서로 분리한다.

## 11.3 컨텍스트는 축약과 확장을 구분한다
- 자세한 버전: README, vision, terminology, rules
- 짧은 버전: GPT Builder instructions, 작업 프롬프트 서두

## 11.4 흔들리면 다시 상위 문서로 돌아간다
작업 중 방향이 흔들리면
새 프롬프트를 더 길게 쓰기보다
project charter, vision, boundaries 같은 상위 문서를 다시 본다.

## 11.5 컨텍스트도 유지보수한다
컨텍스트 문서가 실제 구조와 달라지면
오히려 잘못된 기준이 된다.
따라서 구조 변화가 있으면 컨텍스트 문서도 같이 갱신해야 한다.

---

# 12. 대표 체크리스트

컨텍스트를 정리하거나 갱신할 때 아래를 점검한다.

- 이 정보는 여러 턴에 걸쳐 유효한가?
- 이 정보는 판단을 바꿀 정도로 중요한가?
- 이 정보는 반복 설명을 줄여주는가?
- 이 정보는 README, rules, overview 문서 수준인가?
- 구현 메모가 상시 컨텍스트에 섞여 있지 않은가?
- 현재 실제 구조와 문서가 어긋나지 않는가?

---

# 13. 이 문서의 역할

이 문서는 `office-unify` 프로젝트에서
컨텍스트 엔지니어링을 어떻게 이해하고 운영해야 하는지를 설명하는 기준 문서다.

즉,
- 어떤 정보를 상시 컨텍스트로 볼지
- 어떤 정보는 프롬프트로 처리할지
- 어떤 정보는 계획/보고 문서로 둘지
- Cursor와 GPT Builder에서 각각 어떻게 활용할지

를 정리하는 실무 가이드다.

---

# 14. 결론

이 프로젝트에서 컨텍스트 엔지니어링은
배경 설명을 길게 적는 작업이 아니다.

이 프로젝트에서 컨텍스트 엔지니어링은
**프로젝트의 장기적으로 유효한 원칙, 제약, 우선순위를 구조화하여
AI가 매 턴 흔들리지 않게 만드는 작업**이다.

따라서 `office-unify` 프로젝트에서는
컨텍스트를
- README
- 비전 문서
- 용어집
- 규칙 문서
- 계획 문서

로 나누어 관리하고,
프롬프트는 그 위에서 현재 턴의 목표만 명확히 지시하는 방식으로 운영한다.
