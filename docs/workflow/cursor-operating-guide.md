# Office Unify Cursor Operating Guide

## 문서 목적
이 문서는 `office-unify` 프로젝트를 Cursor에서 실제로 작업할 때
어떤 방식으로 하네스 엔지니어링을 적용할지 설명하는 운영 가이드다.

이 문서는 다음을 다룬다.

- 실제 작업 루트 구성
- Cursor에서의 기본 작업 순서
- 규칙 파일 활용 방식
- 계획 문서 활용 방식
- 프롬프트 운영 방식
- 구현/검증/문서화 흐름
- 피해야 할 작업 방식

즉, 이 문서는
`office-unify-harness`를 실제 Cursor 작업으로 연결하는 실무 운영 문서다.

---

# 1. 기본 원칙

이 프로젝트에서 Cursor는
단순 코드 생성기가 아니라,
하네스 엔지니어링 기반의 작업 보조자 역할을 한다.

따라서 Cursor 작업은 아래 원칙을 따른다.

1. 먼저 분석하고 나중에 구현한다.
2. 큰 작업은 반드시 계획 문서부터 작성한다.
3. 한 번에 작은 단위만 변경한다.
4. 기존 `dev_support` 기능 보호를 최우선으로 둔다.
5. `ai_office`는 참조 원본으로 분석한다.
6. Discord 결합 로직은 직접 이식하지 않는다.
7. 구현 후 검증과 문서화를 함께 수행한다.

---

# 2. 권장 작업 루트 구성

Cursor에서는 멀티 프로젝트를 무작정 따로 여는 것보다,
통합 작업 기준의 상위 폴더 하나를 열고 작업하는 방식을 기본으로 한다.

예시:

```text
D:\workspace\office-unify\
  ai_office\
  dev_support\
  .cursor\
    rules\
    plans\
  docs\
```

핵심은 아래와 같다.

- `ai_office/`는 참조 원본
- `dev_support/`는 실제 통합 대상
- `.cursor/rules/`는 상시 규칙
- `.cursor/plans/`는 작업 계획 문서
- `docs/`는 구조/기록/운영 문서

즉, Cursor는 실제 작업 루트 하나를 열고,
그 안에서 전체 구조를 탐색하도록 하는 것이 좋다.

---

# 3. 실제 작업 시작 전 준비

실제 작업 전에 아래를 준비한다.

## 3.1 코드베이스 준비
- `ai_office` 로컬 체크아웃
- `dev_support` 로컬 체크아웃
- 가능하면 같은 상위 폴더 아래 배치

## 3.2 하네스 파일 반영
`office-unify-harness`의 핵심 파일을 실제 작업 루트에 복사 또는 반영한다.

예:
- `.cursor/rules/00-project-charter.md`
- `.cursor/rules/10-architecture-boundaries.md`
- `.cursor/rules/20-migration-workflow.md`
- `.cursor/rules/30-validation-and-done.md`
- `.cursor/plans/office-unify-master-plan.md`

## 3.3 작업 브랜치 준비
가능하면 `dev_support` 기준으로 통합용 브랜치를 따로 만든다.

예:
- `feature/office-unify-phase1`
- `feature/ai-office-web-migration`

---

# 4. Cursor에서의 기본 작업 순서

이 프로젝트에서 Cursor 작업은 아래 순서를 기본으로 한다.

## 1단계. 분석
관련 파일을 찾고,
현재 구조와 의존 관계를 파악한다.

## 2단계. 설계
범위, 폴더 구조, API, 상태 관리, 리스크를 먼저 정리한다.

## 3단계. 계획 문서 작성
`.cursor/plans/office-unify-master-plan.md` 형식으로
현재 작업 범위를 구조화한다.

## 4단계. 소규모 구현
한 번에 작은 논리 단위만 구현한다.

## 5단계. 검증
lint/build/typecheck/smoke test 등 가능한 검증을 수행한다.

## 6단계. 문서화
README 또는 관련 docs를 갱신한다.

## 7단계. 결과 보고
설계 요약, 변경 파일 목록, 검증, 리스크를 정리한다.

---

# 5. Cursor에서 규칙 파일을 다루는 방식

`.cursor/rules/`의 문서는
작업 전마다 다시 길게 설명하지 않아도 되도록
프로젝트의 상시 기준을 고정하는 역할을 한다.

현재 핵심 규칙은 아래와 같다.

## `00-project-charter.md`
- 프로젝트의 최상위 목적
- 수정 우선순위
- 금지 사항
- 결과 보고 형식

## `10-architecture-boundaries.md`
- 클라이언트/서버 경계
- 순수 로직과 Discord 결합 로직 구분
- 서버 계층 원칙
- 보안상 주의점

## `20-migration-workflow.md`
- 분석 → 설계 → 구현 → 검증 → 문서화 흐름
- 변경 크기 제어
- 단계별 우선순위

## `30-validation-and-done.md`
- 완료 기준
- 검증 기준
- Partial Done / Done 구분
- 결과 보고 태도

Cursor 작업 중에는 이 규칙을 반복 프롬프트로 길게 다시 쓰기보다,
현재 턴의 작업 목표만 명확히 주는 것이 좋다.

---

# 6. 계획 문서 활용 방식

큰 작업에서는 바로 구현하지 않고,
먼저 계획 문서를 채우는 방식을 기본으로 한다.

예시 파일:
- `.cursor/plans/office-unify-master-plan.md`

이 문서는 아래를 정리하는 용도다.

- 현재 범위
- 관련 파일
- 재사용 가능한 로직
- Discord 결합 로직
- 목표 구조
- API 설계
- 구현 단계
- 검증 계획
- 리스크

즉,
계획 문서는 “생각 정리용 메모”가 아니라
실제 구현 전 구조적 기준점이다.

---

# 7. 권장 프롬프트 운영 방식

이 프로젝트에서는 한 번에 모든 것을 시키지 않는다.

권장 프롬프트는 아래처럼 분리한다.

## 7.1 분석 전용 프롬프트
목적:
- 현재 구조를 파악
- 관련 파일 식별
- 재사용 가능 로직/분리 필요 로직 분류

예시:
- 지금은 코드 수정하지 말고 분석만 하라.
- `ai_office`와 `dev_support`를 함께 스캔하고
  재사용 가능한 순수 로직, Discord 결합 로직, 즉시 이관 후보를 분류하라.

---

## 7.2 설계 전용 프롬프트
목적:
- 구조와 범위를 먼저 정리
- 폴더 구조, API, 상태 전략, UI 흐름 결정

예시:
- 현재 분석 결과를 바탕으로
  `.cursor/plans/office-unify-master-plan.md` 형식으로 통합 설계를 작성하라.

---

## 7.3 구현 전용 프롬프트
목적:
- 가장 안전한 1차 변경만 반영
- 작은 단위 구현

예시:
- 기존 기능을 깨지 않는 범위에서
  AI Office 메뉴/레이아웃 추가만 먼저 구현하라.
- 변경 후 lint/build 기준 검증 방법도 정리하라.

---

## 7.4 검증/문서화 프롬프트
목적:
- 변경 후 상태를 명확히 정리
- README/docs 반영 여부 점검
- 리스크 공개

예시:
- 방금 변경 기준으로
  검증 방법, 미검증 영역, 문서 반영 필요 파일을 정리하라.

---

# 8. 권장 작업 단위

좋은 작업 단위는 아래와 같다.

- 상단 내비게이션에 AI Office 메뉴 추가
- `/ai-office` 라우트 골격 추가
- 서버 폴더 구조 초안 추가
- 포트폴리오 조회 API 1개 설계
- AI 토론 서비스 인터페이스 정리
- 트렌드 분석 결과 DTO 정의
- 운영 상태 읽기 페이지 초안

좋지 않은 작업 단위는 아래와 같다.

- 전체 프로젝트 구조 전면 개편
- 모든 Discord 흐름 일괄 이전
- UI/DB/서버/LLM을 한 번에 수정
- 기존 기능 전체 교체
- 분석 없이 대규모 리팩토링

---

# 9. Cursor에게 항상 요구해야 하는 출력 형식

주요 작업 결과는 항상 아래 형식으로 정리하도록 요구한다.

1. 설계 요약
2. 변경 파일 목록
3. 핵심 변경 이유
4. 검증 방법
5. 남은 리스크 / 다음 단계

필요하면 아래를 추가한다.

- 예상 영향 범위
- 롤백 포인트
- 아직 구현하지 않은 부분
- 추가로 검토할 구조 이슈

---

# 10. 검증 명령 운영 원칙

검증은 작업 종류에 따라 달라질 수 있지만,
가능하면 아래 항목을 기준으로 본다.

## 기본 검증 예시
- `npm run lint`
- `npm run build`
- `npm run typecheck`

## 기능 검증 예시
- 기존 업무 도우미 주요 화면 진입
- 신규 메뉴 진입
- 신규 라우트 렌더링
- API 응답 구조 확인
- 에러 상태 확인

## 검증 시 주의
- 실행하지 않은 검증은 했다고 말하지 않는다.
- 환경이 없어 못한 검증은 명시한다.
- 실패한 검증은 숨기지 않는다.

---

# 11. Cursor 작업 시 금지 패턴

아래 방식은 이 프로젝트에서 지양한다.

## 11.1 분석 없이 바로 구현
관련 파일도 충분히 보지 않고 바로 수정하는 방식

## 11.2 한 번에 너무 큰 변경
검토와 롤백이 어려운 수준의 대규모 수정

## 11.3 규칙 무시
`dev_support` 중심 통합, 기존 기능 보호, Discord 직접 이식 금지 등을 무시하는 방식

## 11.4 검증 없는 완료 선언
실행/검토 없이 “완료”라고 말하는 방식

## 11.5 문서 누락
구조가 바뀌었는데 README/docs가 따라오지 않는 방식

---

# 12. 실제 추천 작업 흐름 예시

아래는 실제로 Cursor에서 통합 작업을 할 때의 추천 흐름 예시다.

## 예시 흐름 1. 분석 시작
- 작업 루트를 연다.
- `ai_office`와 `dev_support`를 함께 스캔하게 한다.
- 코드 수정 없이 분석만 지시한다.

## 예시 흐름 2. 설계 정리
- 분석 결과를 바탕으로
  `.cursor/plans/office-unify-master-plan.md`를 채우게 한다.
- 폴더 구조, API, 상태 전략을 정리한다.

## 예시 흐름 3. 1차 구현
- 가장 작은 범위부터 시킨다.
- 예: 메뉴 추가, 라우트 추가, 서버 골격 추가

## 예시 흐름 4. 검증 및 문서화
- lint/build/typecheck 제안 또는 수행
- 관련 문서 갱신
- 리스크와 다음 단계 정리

# 13. Cursor와 GPT Builder의 역할 차이

이 프로젝트에서 Cursor와 GPT Builder는 역할이 다르다.

## Cursor
- 실제 코드베이스 분석
- 파일 수정
- 구조 리팩토링
- 구현/검증 지원

## GPT Builder
- 상위 원칙 저장
- 지침 요약
- 문서/판단 기준 정리
- 설명형 보조

즉,
Cursor는 실행 도구에 가깝고,
GPT Builder는 지식/지침 저장소에 가깝다.

실제 코드 통합 하네스의 중심은 Cursor 쪽에 둔다.

---

# 14. 이 문서의 역할

이 문서는
`office-unify-harness`의 규칙과 계획을
실제 Cursor 작업 순서로 연결하는 운영 문서다.

이 문서를 통해
- 실제 작업 루트를 어떻게 구성할지 정하고
- 어떤 순서로 작업할지 고정하며
- 어떤 프롬프트 흐름이 적절한지 정리하고
- 어떤 출력 형식을 강제할지 명확히 한다.

---

# 15. 결론

이 프로젝트에서 Cursor는
즉흥적으로 코드를 바꾸는 도구가 아니라,
하네스 엔지니어링 체계 안에서 움직이는 실행 보조자다.

따라서 실제 작업은 다음을 항상 포함해야 한다.

- 규칙 파일 반영
- 계획 문서 작성
- 분석 선행
- 작은 단위 구현
- 검증 수행
- 문서 갱신
- 리스크 공개

이 문서는 그 운영 흐름의 기준점이다.
