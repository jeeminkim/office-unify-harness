# Office Unify Harness Engineering Guide

## 문서 목적
이 문서는 `office-unify` 프로젝트에서 하네스 엔지니어링을 어떻게 적용하고 운영할지 설명하는 실무 가이드다.

이 문서는 단순 개념 설명이 아니라,
실제 Cursor 작업과 문서 관리, 계획 수립, 검증, 결과 보고까지 포함한
프로젝트 운영 기준을 정리하기 위해 존재한다.

---

# 1. 하네스 엔지니어링의 정의

이 프로젝트에서 하네스 엔지니어링은
“AI에게 좋은 프롬프트를 한 번 쓰는 것”이 아니다.

이 프로젝트에서 하네스 엔지니어링은
AI가 일관되게 좋은 결과를 내도록
작업 전체의 레일을 설계하는 것을 뜻한다.

즉, 아래를 함께 다룬다.

- 프로젝트 원칙
- 아키텍처 경계
- 작업 순서
- 계획 문서
- 검증 기준
- 완료 기준
- 결과 보고 형식
- 리스크 기록
- 문서 업데이트 기준

이 프로젝트에서는 프롬프트 엔지니어링보다
하네스 엔지니어링이 상위 개념이다.

---

# 2. 왜 이 프로젝트에 하네스가 필요한가

`office-unify` 프로젝트는 단순 기능 추가 프로젝트가 아니다.

동시에 아래가 필요하다.

- 두 코드베이스 구조 이해
- `ai_office`와 `dev_support`의 역할 구분
- Discord 결합 로직 분리
- 웹 구조로의 재설계
- 서버/클라이언트 경계 유지
- 보안 관리
- 점진적 마이그레이션
- 기존 기능 보호
- 문서화 및 검증

이런 프로젝트를 프롬프트만으로 운영하면
매 턴마다 기준이 흔들리거나,
너무 큰 변경이 발생하거나,
검증 없는 완료 선언이 나올 가능성이 높다.

그래서 이 프로젝트는
규칙과 계획을 먼저 세우고,
그 위에서 프롬프트를 사용하는 방식으로 운영한다.

---

# 3. 하네스의 구성 요소

이 프로젝트의 하네스는 크게 5개 층으로 구성된다.

## 3.1 프로젝트 규칙
프로젝트 전체에 항상 적용되는 원칙이다.

예:
- `.cursor/rules/00-project-charter.md`
- `.cursor/rules/10-architecture-boundaries.md`
- `.cursor/rules/20-migration-workflow.md`
- `.cursor/rules/30-validation-and-done.md`

이 규칙들은
무엇이 허용되고 무엇이 금지되는지,
어떤 순서로 작업해야 하는지,
무엇을 완료로 볼지 정한다.

---

## 3.2 계획 문서
실제 작업 전에 범위와 단계, 리스크를 정리하는 문서다.

예:
- `.cursor/plans/office-unify-master-plan.template.md`

계획 문서는
“지금 무엇을 할지”를 구체화하는 장치다.

---

## 3.3 작업 프롬프트
현재 턴에서 AI가 해야 할 일을 직접 지시하는 문장이다.

예:
- 분석 전용 프롬프트
- 설계 전용 프롬프트
- 구현 전용 프롬프트
- 문서화/검증 전용 프롬프트

프롬프트는 필요하지만,
규칙과 계획 없이 단독으로 쓰지 않는다.

---

## 3.4 검증 기준
작업 결과가 실제로 신뢰 가능한지 판단하는 기준이다.

예:
- lint
- build
- typecheck
- smoke test
- 기존 기능 회귀 여부
- 문서 반영 여부

---

## 3.5 결과 보고 형식
AI가 어떤 형식으로 결과를 제출해야 하는지 정하는 기준이다.

기본 형식:
1. 설계 요약
2. 변경 파일 목록
3. 핵심 변경 이유
4. 검증 방법
5. 남은 리스크 / 다음 단계

---

# 4. 프롬프트 엔지니어링과의 관계

하네스 엔지니어링은 프롬프트 엔지니어링을 대체하는 것이 아니라,
그 위에서 프롬프트가 안정적으로 작동하게 만드는 구조다.

## 프롬프트 엔지니어링
현재 턴에서 무엇을 시킬지 정한다.

예:
- “지금은 분석만 하라.”
- “코드 수정 없이 설계만 제안하라.”
- “가장 안전한 1차 변경만 구현하라.”

## 하네스 엔지니어링
프롬프트가 잘못된 방향으로 과도하게 해석되지 않도록
프로젝트의 전체 레일을 만든다.

예:
- `dev_support`가 최종 본체라는 점
- Discord 결합 로직 직접 이식 금지
- 분석 → 설계 → 구현 → 검증 → 문서화 순서
- 완료 기준과 결과 보고 형식

즉,
프롬프트는 방향 지시이고,
하네스는 탈선 방지 장치다.

---

# 5. 컨텍스트 엔지니어링과의 관계

컨텍스트 엔지니어링은
AI가 지속적으로 알고 있어야 하는 배경 지식을 구조화하는 활동이다.

이 프로젝트에서 대표적인 컨텍스트는 아래와 같다.

- `dev_support`가 최종 제품 본체
- `ai_office`는 참조 원본
- Discord UI는 웹 구조로 치환
- 기존 기능 보호 우선
- 민감정보는 서버 전용
- 점진적 통합 우선

하네스 엔지니어링은
이런 컨텍스트를 단순 설명으로 끝내지 않고,
규칙 파일, 계획 문서, 체크리스트로 고정한다.

즉,
컨텍스트 엔지니어링은 배경을 만들고,
하네스 엔지니어링은 그 배경을 운영 구조로 굳힌다.

---

# 6. 실제 운영 순서

이 프로젝트에서 하네스 엔지니어링은 아래 순서로 작동한다.

## 1단계. 상위 원칙 확정
프로젝트 헌장과 아키텍처 경계를 먼저 만든다.

관련 문서:
- `00-project-charter.md`
- `10-architecture-boundaries.md`

---

## 2단계. 실행 워크플로 확정
작업 순서와 완료 기준을 만든다.

관련 문서:
- `20-migration-workflow.md`
- `30-validation-and-done.md`

---

## 3단계. 계획 템플릿 준비
실제 작업 전에 어떤 형식으로 계획을 세울지 준비한다.

관련 문서:
- `office-unify-master-plan.template.md`

---

## 4단계. 실제 작업 루트에 반영
하네스 저장소의 핵심 파일을
실제 Cursor 작업 루트에 반영한다.

예:
- `.cursor/rules/`
- `.cursor/plans/`
- `docs/`

---

## 5단계. 작업 프롬프트 사용
분석, 설계, 구현, 검증을 한 번에 몰아 하지 않고,
작업 목적별로 프롬프트를 나누어 사용한다.

---

## 6단계. 검증 및 보고
모든 주요 작업은
검증과 문서화, 결과 보고까지 포함해 마무리한다.

---

# 7. 이 프로젝트의 기본 운영 모드

이 프로젝트는 아래 모드를 기본으로 한다.

## 분석 우선 모드
관련 파일을 먼저 찾고,
바로 구현하지 않는다.

## 설계 선행 모드
구현 전에 구조와 범위를 먼저 정한다.

## 소규모 구현 모드
한 번에 한 기능 또는 한 논리 단위만 다룬다.

## 검증 포함 모드
작업 결과는 가능한 한 검증한다.

## 문서 동반 모드
구조 변경 시 문서를 같이 갱신한다.

즉, 이 프로젝트는
“바로 만들어보는 방식”보다
“작게 설계하고 검증하며 이동하는 방식”을 기본으로 삼는다.

---

# 8. Cursor에서의 적용 방식

Cursor는 실제 작업 루트의 코드베이스와 규칙 파일을 함께 활용한다.

따라서 이 저장소의 문서만 만들어두고 끝내면 충분하지 않다.
실제 작업 시에는 아래가 필요하다.

- 실제 코드베이스 열기
- `.cursor/rules/` 반영
- `.cursor/plans/` 반영
- 분석 전용 프롬프트 사용
- 계획 문서 작성
- 작은 단위 구현
- 검증 및 보고

즉,
하네스 저장소는 원본이고,
실제 작업 루트는 실행 환경이다.

---

# 9. 권장 작업 단위

이 프로젝트에서 권장하는 작업 단위는 아래와 같다.

좋은 작업 단위 예시:
- AI Office 메뉴 추가
- 서버 폴더 골격 추가
- 포트폴리오 조회 API 1개 추가
- AI 토론 서비스 추출
- 트렌드 분석 실행 흐름 1개 연결
- 운영 상태 읽기 페이지 초안 작성

피해야 할 작업 단위 예시:
- 전체 라우팅 전면 개편
- Discord 로직 일괄 이전
- UI/서버/DB 구조 동시 전면 수정
- 기존 기능을 한 번에 대체하는 대규모 변경

---

# 10. 하네스가 방지하려는 대표 문제

이 프로젝트에서 하네스는 아래 문제를 줄이기 위해 존재한다.

## 10.1 과도한 범위 확대
한 번의 프롬프트로 너무 많은 작업을 시도하는 문제

## 10.2 분석 없는 구현
구조 파악 없이 바로 코드부터 고치는 문제

## 10.3 경계 붕괴
클라이언트/서버, 순수 로직/Discord 로직 경계가 흐려지는 문제

## 10.4 검증 없는 완료 선언
동작 확인 없이 완료라고 말하는 문제

## 10.5 문서 누락
실제 구조가 바뀌었는데 문서가 따라오지 않는 문제

## 10.6 리스크 은폐
불확실하거나 미검증된 상태를 숨기고 진행하는 문제

---

# 11. 향후 확장 방향

하네스는 현재 규칙 파일과 계획 템플릿에서 시작하지만,
향후 아래로 확장할 수 있다.

- 분석 전용 프롬프트 모음
- 구현 전용 프롬프트 모음
- 디버깅 프롬프트
- 체크리스트 템플릿
- 리스크 로그 템플릿
- 결과 보고 템플릿
- GPT Builder용 요약 지침
- 팀 협업용 운영 가이드

즉,
하네스는 정적인 문서 세트가 아니라,
프로젝트 운영 품질을 높이기 위한 확장 가능한 체계다.

---

# 12. 이 문서의 역할

이 문서는
`office-unify` 프로젝트에서 하네스 엔지니어링을 어떤 뜻으로 쓰는지,
그리고 실제로 어떻게 운영하는지 설명하는 기준 문서다.

이 문서를 통해
- 하네스의 필요성을 공유하고
- 각 문서의 역할을 이해하며
- 실제 Cursor 작업 순서를 정렬하고
- 프로젝트 운영 방식을 일관되게 유지한다.

---

# 13. 결론

이 프로젝트에서 하네스 엔지니어링은 선택 사항이 아니다.

이 통합은
- 구조적 복잡성이 높고
- 기존 기능 보호가 중요하며
- 점진적 마이그레이션이 필요하고
- 보안, 검증, 문서화가 동시에 요구되기 때문이다.

따라서 이 프로젝트는
즉흥적인 프롬프트 기반 개발이 아니라,
**규칙 + 계획 + 검증 + 보고를 포함한 하네스 엔지니어링 방식**으로 운영한다.

이 문서는 그 운영 방식의 실무 가이드다.
