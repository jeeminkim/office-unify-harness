# Office Unify Prompt Engineering Guide

## 문서 목적
이 문서는 `office-unify` 프로젝트에서
프롬프트 엔지니어링을 어떻게 이해하고 사용할지 정리하는 실무 가이드다.

이 프로젝트에서 프롬프트는 중요하지만,
프롬프트만으로 프로젝트를 운영하지 않는다.

따라서 이 문서는
- 프롬프트의 역할
- 프롬프트가 필요한 시점
- 프롬프트를 어떻게 나눠야 하는지
- 어떤 프롬프트가 좋은 프롬프트인지
- 하네스 엔지니어링과 어떤 관계인지

를 설명한다.

---

# 1. 이 프로젝트에서 프롬프트 엔지니어링이란

이 프로젝트에서 프롬프트 엔지니어링은
“AI에게 원하는 답을 잘 뽑아내는 문장 기술”에만 머무르지 않는다.

이 프로젝트에서 프롬프트 엔지니어링은
**현재 턴에서 AI가 해야 할 작업 범위와 작업 태도를 명확히 지정하는 것**을 뜻한다.

즉, 프롬프트는 아래를 결정한다.

- 지금 분석만 할지
- 설계만 할지
- 작은 구현만 할지
- 검증과 문서화만 할지
- 무엇을 포함하고 무엇을 하지 말아야 할지

프롬프트는 “이번 턴의 작업 계약서”에 가깝다.

---

# 2. 왜 프롬프트가 중요한가

`office-unify` 프로젝트는 다음과 같은 특성이 있다.

- 두 저장소를 함께 봐야 한다
- 구조적 판단이 많다
- 기존 기능 보호가 중요하다
- Discord 결합 로직을 분리해야 한다
- 웹 전환 판단이 필요하다
- 보안 경계를 유지해야 한다
- 단계별 마이그레이션이 필요하다

이런 프로젝트에서 프롬프트가 모호하면
아래 문제가 쉽게 발생한다.

- 분석 없이 바로 구현
- 한 번에 너무 큰 변경
- 기존 기능 파손
- Discord 로직의 무분별한 복사
- 검증 없는 완료 선언
- 문서화 누락

따라서 프롬프트는
작업 범위를 좁히고,
현재 단계의 목표를 분명히 하며,
하네스 규칙을 이번 턴에 맞게 활성화하는 역할을 한다.

---

# 3. 하네스 엔지니어링과의 관계

이 프로젝트에서는 프롬프트 엔지니어링이 하네스 엔지니어링보다 상위 개념이 아니다.

오히려 다음 관계로 본다.

## 하네스 엔지니어링
- 프로젝트 전체의 레일
- 규칙
- 계획
- 검증 기준
- 완료 기준
- 결과 보고 형식

## 프롬프트 엔지니어링
- 현재 턴의 작업 지시
- 현재 단계의 범위 제한
- 지금 필요한 산출물 지정
- 금지 사항 재강조

즉,
하네스는 상시 기준이고,
프롬프트는 현재 턴의 실행 지시다.

프롬프트는 하네스를 대체하지 않는다.
프롬프트는 하네스를 현재 작업에 적용하는 도구다.

---

# 4. 좋은 프롬프트의 조건

이 프로젝트에서 좋은 프롬프트는
예쁘게 쓰인 프롬프트가 아니라,
**작업이 흔들리지 않도록 범위와 기대 결과가 명확한 프롬프트**다.

좋은 프롬프트는 아래 조건을 갖는다.

## 4.1 현재 단계가 분명하다
예:
- 지금은 분석만
- 지금은 설계만
- 지금은 가장 안전한 1차 구현만
- 지금은 검증과 문서화만

## 4.2 범위가 제한되어 있다
예:
- 코드 수정 금지
- 한 번에 작은 변경만
- 기존 기능 교체 금지
- Discord 로직 직접 이식 금지

## 4.3 프로젝트 원칙이 반영되어 있다
예:
- `dev_support` 중심 통합
- `ai_office`는 참조 원본
- 기존 기능 보호
- 서버 전용 민감정보 처리

## 4.4 기대 출력 형식이 정해져 있다
예:
1. 설계 요약
2. 변경 파일 목록
3. 핵심 변경 이유
4. 검증 방법
5. 남은 리스크 / 다음 단계

## 4.5 금지 사항이 명시되어 있다
예:
- 분석 없이 구현 금지
- 검증 없는 완료 선언 금지
- 대규모 일괄 수정 금지

---

# 5. 나쁜 프롬프트의 특징

이 프로젝트에서 피해야 할 프롬프트는 아래와 같다.

## 5.1 너무 넓은 프롬프트
예:
- “전체 통합 다 해줘”
- “좋은 방향으로 알아서 다 바꿔줘”

문제:
- 범위가 무한정 커짐
- AI가 임의 판단을 과하게 함
- 검토/롤백이 어려움

## 5.2 단계가 섞인 프롬프트
예:
- 분석도 하고 설계도 하고 구현도 하고 문서도 다 해줘

문제:
- 분석 없이 구현이 섞임
- 품질 기준이 흐려짐
- 결과물이 모호해짐

## 5.3 결과 형식이 없는 프롬프트
예:
- “한번 정리해줘”
- “좋은 방향 알려줘”

문제:
- 출력 품질이 들쭉날쭉함
- 후속 작업으로 연결이 어려움

## 5.4 금지 사항이 없는 프롬프트
문제:
- 기존 기능을 건드릴 위험
- 대규모 구조 변경 위험
- 검증 없이 끝날 위험

---

# 6. 권장 프롬프트 유형

이 프로젝트에서는 프롬프트를 목적별로 나눈다.

## 6.1 분석 전용 프롬프트
목적:
- 관련 파일 식별
- 구조 파악
- 순수 로직 / Discord 결합 로직 분류

핵심 제약:
- 코드 수정 금지
- 파일 생성 금지

핵심 산출물:
- 분석 요약
- 분류 결과
- 예상 변경 파일 범주

---

## 6.2 설계 전용 프롬프트
목적:
- 폴더 구조
- API 구조
- 상태 관리 전략
- 단계별 마이그레이션 계획 정리

핵심 제약:
- 구현 금지
- 설계와 계획만 수행

핵심 산출물:
- 통합 아키텍처 초안
- 마스터 플랜
- 리스크와 검증 계획

---

## 6.3 구현 전용 프롬프트
목적:
- 가장 안전한 범위의 1차 구현
- 회귀 위험이 낮은 범위의 실제 변경

핵심 제약:
- 작은 범위만 구현
- 기존 기능 보호
- 대규모 교체 금지

핵심 산출물:
- 변경 파일 목록
- 구현 이유
- 검증 계획 또는 검증 결과

---

## 6.4 검증/문서화 프롬프트
목적:
- 변경 검증
- 문서 반영 필요성 판단
- Done / Partial Done 판정

핵심 제약:
- 검증하지 않은 것을 했다고 말하지 않기
- 리스크를 숨기지 않기

핵심 산출물:
- 검증 결과
- 미검증 영역
- 문서 반영 대상
- 상태 판정

---

# 7. 프롬프트 작성 기본 구조

이 프로젝트에서는 프롬프트를 아래 구조로 쓰는 것이 좋다.

## 7.1 역할 지정
예:
- Next.js/React 프런트엔드 아키텍트
- Node.js/TypeScript 백엔드 리팩토링 전문가

## 7.2 프로젝트 배경
예:
- `ai_office`와 `dev_support` 통합
- `dev_support` 중심 통합 + `ai_office` 서버 로직 흡수

## 7.3 현재 턴 목표
예:
- 지금은 분석만
- 지금은 설계만
- 지금은 가장 안전한 1차 구현만

## 7.4 반드시 따를 원칙
예:
- 기존 기능 보호
- Discord 직접 이식 금지
- 민감정보 서버 전용

## 7.5 구체적 작업 항목
예:
- 순수 로직 후보 목록
- 예상 폴더 구조
- API 후보
- 예상 변경 파일

## 7.6 출력 형식
예:
1. 설계 요약
2. 변경 파일 목록
3. 핵심 변경 이유
4. 검증 방법
5. 남은 리스크 / 다음 단계

## 7.7 금지 사항
예:
- 코드 수정 금지
- 대규모 변경 금지
- 검증 없는 완료 선언 금지

---

# 8. 실제 운영 원칙

프롬프트는 한 번에 모든 것을 해결하려고 하지 않는다.

권장 운영 방식은 아래와 같다.

1. 분석 프롬프트
2. 설계 프롬프트
3. 구현 프롬프트
4. 검증/문서화 프롬프트

즉,
작업을 턴 단위로 쪼개고,
각 턴마다 하나의 목적만 강조한다.

이렇게 해야
- 범위가 작아지고
- 결과가 명확해지고
- 품질 관리가 쉬워진다.

---

# 9. 이 프로젝트에 맞는 프롬프트 톤

이 프로젝트에서 프롬프트는
막연하게 창의적인 답변을 유도하기보다,
**정확하고 제한적이며 검증 지향적으로** 작성하는 편이 좋다.

권장 톤:
- 구체적
- 단계적
- 제한적
- 검증 중심
- 리스크 공개형

지양할 톤:
- 전부 알아서
- 최대한 많이
- 좋은 방향으로
- 필요하면 마음대로

---

# 10. 프롬프트만으로 해결하려 하지 말 것

아무리 좋은 프롬프트라도
아래를 대체할 수 없다.

- 프로젝트 헌장
- 아키텍처 경계 문서
- 마이그레이션 워크플로
- 완료 기준
- 계획 문서
- 결과 보고 템플릿

즉,
프롬프트는 중요하지만,
프롬프트만 강하게 만들면 해결된다고 생각하면 안 된다.

이 프로젝트는
프롬프트 + 컨텍스트 + 하네스가 함께 가야 한다.

---

# 11. 대표 체크리스트

프롬프트를 쓰기 전에 아래를 점검한다.

- 지금 단계가 분석인가, 설계인가, 구현인가, 검증인가?
- 이번 턴에서 절대 하지 말아야 할 것은 무엇인가?
- 기존 기능 보호 조건이 포함되었는가?
- Discord 직접 이식 금지가 포함되었는가?
- 출력 형식이 정해져 있는가?
- 검증 또는 미검증 구분을 요구하고 있는가?
- 범위가 너무 크지 않은가?

---

# 12. 이 문서의 역할

이 문서는 `office-unify` 프로젝트에서
프롬프트를 어떻게 써야 하는지에 대한 기준 문서다.

즉,
- 어떤 프롬프트가 좋은지
- 어떤 프롬프트가 위험한지
- 프롬프트를 어떤 목적별로 나눠야 하는지
- 하네스와 어떤 관계인지

를 정리하는 운영 가이드다.

---

# 13. 결론

이 프로젝트에서 프롬프트 엔지니어링은
AI에게 멋진 말을 하는 기술이 아니다.

이 프로젝트에서 프롬프트 엔지니어링은
**현재 턴의 작업을 안전하게 제한하고,
기대 결과를 명확히 하고,
하네스 규칙을 현재 단계에 맞게 적용하는 기술**이다.

따라서 `office-unify` 프로젝트에서는
프롬프트를 항상
- 단계별로 나누고
- 범위를 제한하고
- 출력 형식을 고정하고
- 금지 사항을 함께 명시하는 방식으로 사용한다.
