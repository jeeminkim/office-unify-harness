# Cursor Prompt - Design and Master Plan

당신은 Next.js/React 프런트엔드 아키텍트이자 Node.js/TypeScript 백엔드 리팩토링 전문가다.

현재 작업은 `ai_office`와 `dev_support`를 하나의 통합 웹 서비스로 재구성하는 프로젝트이며,
최종 방향은 **`dev_support` 중심 통합 + `ai_office` 서버 로직 흡수**다.

현재 작업 루트에는 아래 두 저장소가 함께 있다고 가정한다.

- `ai_office/` : 기존 Discord 기반 AI Office 서비스 원본 참조용
- `dev_support/` : 현재 React/Next.js 기반 업무 도우미 앱, 통합의 최종 타깃

중요:
- 지금은 **코드 수정하지 마라**
- 지금은 **설계와 계획 수립만 수행하라**
- 파일 생성, 삭제, 코드 패치, 리팩토링은 하지 마라
- 이번 턴의 목적은 **통합 설계안과 마스터 플랜 작성**이다

---

## 반드시 따를 프로젝트 원칙

1. 최종 제품 본체는 `dev_support`다.
2. `ai_office`는 기능 원천이자 참조 원본이다.
3. 기존 `dev_support` 기능은 깨지면 안 된다.
4. Discord 결합 로직은 웹에 직접 이식하지 않는다.
5. 순수 비즈니스 로직은 가능한 한 재사용 후보로 본다.
6. 민감정보, API Key, Supabase service role key 등은 서버 전용으로 유지해야 한다.
7. 통합은 항상 분석 → 설계 → 구현 → 검증 → 문서화 순서로 진행한다.
8. 지금 단계에서는 구현이 아니라 설계와 계획 고도화가 목표다.

---

## 전제

이전 단계에서 아래가 어느 정도 분석되었다고 가정한다.

- `dev_support`의 핵심 기능과 유지 범위
- `ai_office`의 핵심 기능과 Discord 결합 영역
- 순수 로직 / 서버 전용 로직 / Discord 결합 로직 / 운영 전용 로직 분류
- 웹 전환 가치가 높은 기능 후보
- 1차 MVP 우선순위 후보

이제 그 분석을 바탕으로
**실제 구현 가능한 수준의 통합 설계와 단계별 마스터 플랜**을 작성하라.

## 이번 설계의 목표

아래를 중심으로 설계를 정리하라.

### 1. 통합 아키텍처 초안
- `dev_support` 중심 구조에서 `ai_office` 기능을 어떻게 흡수할지
- 어떤 계층을 새로 만들지
- 어떤 로직을 서버 계층으로 이동시킬지
- 어떤 로직은 legacy adapter로 남겨야 할지

### 2. 폴더 구조 제안
- 실제 구현 가능한 수준으로 제안하라
- 클라이언트/서버 경계를 반영하라
- provider / service / repository / contract 분리를 고려하라

### 3. 라우팅 / IA 설계
- 통합 홈
- AI Office 섹션
- 기존 업무 도우미 섹션
- 공통 레이아웃/내비게이션/상태 패널 방향

### 4. API 설계 초안
- 1차 MVP에 필요한 API 후보
- 입력/출력/보안상 고려점
- 서버 의존성과 데이터 흐름

### 5. 상태 관리 전략
- 클라이언트 상태와 서버 상태 구분
- source of truth를 어디에 둘지
- localStorage 의존 범위 제한

### 6. 점진적 마이그레이션 계획
- 한 번에 모두 옮기지 말고
- Phase 단위로 작은 변경 묶음으로 나눠라
- 각 단계별 목표와 완료 조건을 명확히 하라

### 7. 검증 계획
- 어떤 검증을 어떤 단계에서 수행할지
- 기존 기능 회귀를 어떻게 확인할지
- 문서 갱신 범위를 어떻게 관리할지

---

## 반드시 반영할 통합 방향

아래 방향을 설계안에 포함하라.

### 프런트엔드 방향
- `dev_support`의 기존 React/Next.js 구조와 UI 자산을 최대한 재사용
- 공통 레이아웃과 네비게이션을 우선 정비
- `AI Office`와 `업무 도우미`를 하나의 앱 안에서 제공
- 예시 라우트:
  - `/`
  - `/ai-office`
  - `/ai-office/portfolio`
  - `/ai-office/debate`
  - `/ai-office/trends`
  - `/ai-office/ops`
  - `/work-helper`

### 서버 방향
- `ai_office`에서 Discord 결합 로직을 제외한 핵심 로직을 서버 계층으로 흡수
- 예시 구조:
  - `src/server/ai-office/contracts/`
  - `src/server/ai-office/providers/`
  - `src/server/ai-office/services/`
  - `src/server/ai-office/repositories/`
  - `src/server/ai-office/utils/`
  - `app/api/ai-office/*`

### 1차 MVP 범위
이번 설계는 아래를 우선 범위로 본다.

1. 기존 `dev_support` 기능 유지
2. AI Office 메뉴/레이아웃 추가
3. 웹 전환 가치가 높은 핵심 기능 우선 흡수
   - 포트폴리오 조회
   - AI 토론 실행
   - 트렌드 분석 실행
4. 운영/제어 기능은 읽기 중심 상태 페이지부터 시작
5. Discord 전용 운영 기능은 즉시 제거하지 않고 legacy adapter 후보로 본다

## 설계 방식

다음 순서로 설계하라.

1. 현재 분석 결과를 짧게 요약하라
2. 통합 제품의 최상위 구조를 정의하라
3. 폴더 구조를 제안하라
4. 클라이언트/서버 경계를 정리하라
5. API와 상태 관리 방향을 제안하라
6. 1차 MVP 범위와 제외 범위를 명시하라
7. Phase 단위의 마이그레이션 계획을 작성하라
8. 예상 변경 파일과 리스크를 정리하라
9. 검증 및 문서화 계획을 포함하라
