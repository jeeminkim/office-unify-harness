# Cursor Prompt - Safe First Implementation

당신은 Next.js/React 프런트엔드 아키텍트이자 Node.js/TypeScript 백엔드 리팩토링 전문가다.

현재 작업은 `ai_office`와 `dev_support`를 하나의 통합 웹 서비스로 재구성하는 프로젝트이며,
최종 방향은 **`dev_support` 중심 통합 + `ai_office` 서버 로직 흡수**다.

현재 작업 루트에는 아래 두 저장소가 함께 있다고 가정한다.

- `ai_office/` : 기존 Discord 기반 AI Office 서비스 원본 참조용
- `dev_support/` : 현재 React/Next.js 기반 업무 도우미 앱, 통합의 최종 타깃

이번 턴의 목적은
**전체 통합이 아니라, 가장 안전한 1차 구현만 수행하는 것**이다.

중요:
- 한 번에 크게 바꾸지 마라
- 기존 `dev_support` 핵심 기능을 절대 깨뜨리지 마라
- 설계 문서와 하네스 규칙을 우선 따르라
- 작은 범위, 검증 가능한 범위, 롤백 쉬운 범위만 구현하라

---

## 반드시 따를 프로젝트 원칙

1. 최종 제품 본체는 `dev_support`다.
2. `ai_office`는 기능 원천이자 참조 원본이다.
3. 기존 `dev_support` 기능은 깨지면 안 된다.
4. Discord 결합 로직은 웹에 직접 이식하지 않는다.
5. 순수 비즈니스 로직은 가능한 한 재사용 후보로 본다.
6. 민감정보, API Key, Supabase service role key 등은 서버 전용으로 유지해야 한다.
7. 통합은 항상 분석 → 설계 → 구현 → 검증 → 문서화 순서로 진행한다.
8. 이번 단계에서는 **가장 안전한 1차 변경만 구현**한다.
9. 대규모 구조 변경, 전면 교체, 일괄 이전은 금지한다.

## 전제

이전 단계에서 아래가 정리되었다고 가정한다.

- 분석 결과가 존재한다
- 통합 아키텍처 초안이 존재한다

- 1차 MVP 범위가 어느 정도 합의되었다

이제 그 설계를 바탕으로
**가장 작은 범위의 1차 구현만 반영하라.**

## 이번 구현의 목표

이번 턴에서는 아래 조건을 만족하는 변경만 수행하라.

### 안전한 1차 구현의 조건
- 기존 `dev_support` 기능 회귀 위험이 낮다
- 기존 코드를 전면 교체하지 않는다
- 구조적으로 다음 단계를 위한 발판이 된다
- 검증 범위가 비교적 명확하다
- UI/서버/DB/LLM을 한 번에 모두 건드리지 않는다

---

## 권장 구현 우선순위

아래 중 **가장 안전한 범위부터** 선택하라.

### 우선순위 예시 1. AI Office 메뉴/레이아웃 추가
- 기존 앱 구조를 유지한 채
- 통합 앱의 상위 네비게이션 또는 섹션 진입점만 추가
- 아직 내부 기능은 더미/placeholder로 둬도 된다

### 우선순위 예시 2. `/ai-office` 라우트 골격 추가
- 페이지 라우트 추가
- 공통 레이아웃 연결
- 내부에는 현재 단계 설명, 준비 중 상태, 카드 UI 정도만 둔다

### 우선순위 예시 3. 서버 폴더 골격 추가
- 아직 실제 복잡한 로직 이관 전이라면
- `src/server/ai-office/*` 구조만 먼저 잡고
- 최소한의 contracts/service/provider 틀만 만든다

### 우선순위 예시 4. 읽기 전용 API 1개 초안 추가
- 가능하다면 회귀 위험이 낮은 API 하나만
- 예: 포트폴리오 조회용 read-only route 초안
- 단, 민감정보/외부 연동/불안정한 의존성은 신중히 다룬다

## 이번 턴에서 피해야 할 구현

아래는 이번 턴에서 하지 마라.

- 모든 Discord 흐름 일괄 이전
- AI 토론/트렌드/포트폴리오를 한 번에 모두 구현
- 기존 `dev_support` 라우팅 전면 개편
- LLM provider 전면 통합
- DB 구조 대규모 변경
- 운영 패널 전체 이식
- 검증 어려운 대규모 리팩토링
- 설계 없이 여러 계층(UI/API/DB/Provider)을 동시에 대폭 수정

- 최소한 어떤 검증이 필요한지 명확히 정리하라

### 5. 문서 반영 필요성 판단

## 구현 시 반드시 지켜야 할 세부 원칙

### 1. 기존 기능 보호
- 기존 업무 도우미의 flow/sql/ts 생성 등 핵심 기능을 깨뜨리지 마라
- 공통 컴포넌트나 기존 페이지 동작에 영향을 주는 변경은 최소화하라

### 2. 추가 방식 우선
- 기존 것을 교체하기보다 새 구조를 덧붙이는 방식으로 접근하라
- 예: 메뉴 추가, 새 라우트 추가, 새 서버 폴더 추가

### 3. 경계 유지
- 클라이언트/서버 경계를 흐리지 마라
- 민감정보/외부 연동은 서버 쪽으로만 유지하라

### 4. Discord 직접 이식 금지
- Discord interaction, panel restore, ephemeral reply, customId 흐름을
  그대로 복사하지 마라
- 필요하면 나중 단계에서 웹 치환 또는 legacy adapter 방향으로 분리하라

### 5. 과도한 추정 금지
- 현재 코드베이스에서 확인한 것과 추정한 것을 구분하라
- 검증하지 못한 사항은 숨기지 마라
