# Cursor Prompt - Verification and Documentation

당신은 Next.js/React 프런트엔드 아키텍트이자 Node.js/TypeScript 백엔드 리팩토링 전문가다.

현재 작업은 `ai_office`와 `dev_support`를 하나의 통합 웹 서비스로 재구성하는 프로젝트이며,
최종 방향은 **`dev_support` 중심 통합 + `ai_office` 서버 로직 흡수**다.

현재 작업 루트에는 아래 두 저장소가 함께 있다고 가정한다.

- `ai_office/` : 기존 Discord 기반 AI Office 서비스 원본 참조용
- `dev_support/` : 현재 React/Next.js 기반 업무 도우미 앱, 통합의 최종 타깃

이번 턴의 목적은
**최근 변경 사항을 기준으로 검증 가능 범위를 점검하고,
문서 반영 필요 여부를 판단하며,
결과를 완료 기준에 맞게 정리하는 것**이다.

중요:
- 이번 턴은 구현 자체보다 **검증과 문서화**가 중심이다
- 검증하지 않은 것을 검증했다고 말하지 마라
- 문서 반영이 필요하면 구체적으로 지적하라
- 미완료 상태를 완료처럼 포장하지 마라

---

## 반드시 따를 프로젝트 원칙

1. 최종 제품 본체는 `dev_support`다.
2. `ai_office`는 기능 원천이자 참조 원본이다.
3. 기존 `dev_support` 기능은 깨지면 안 된다.
4. Discord 결합 로직은 웹에 직접 이식하지 않는다.
5. 민감정보, API Key, Supabase service role key 등은 서버 전용으로 유지해야 한다.
6. 통합은 항상 분석 → 설계 → 구현 → 검증 → 문서화 순서로 진행한다.
7. 완료 선언은 반드시 검증 근거와 함께 해야 한다.
8. 문서와 리스크 공개는 구현과 동일한 수준의 필수 산출물이다.

## 이번 턴의 목표

아래를 수행하라.

### 1. 최근 변경 사항 정리
- 무엇이 바뀌었는지 요약하라
- 어떤 파일이 변경되었는지 정리하라
- 변경의 목적을 설명하라

### 2. 검증 가능 항목 식별
- 현재 변경 범위 기준으로 어떤 검증이 가능한지 구분하라
- lint/build/typecheck 등 공통 검증 후보를 제시하라
- 기능별 smoke test 후보를 제시하라
- 기존 기능 회귀 위험 포인트를 짚어라

### 3. 검증 수행 또는 검증 계획 정리
- 실제 수행 가능한 검증은 수행하라
- 수행하지 못한 검증은 왜 못했는지 적어라
- 환경 의존적인 항목은 별도로 분리하라

### 4. 문서 반영 필요성 판단
- README 갱신이 필요한지 판단하라
- docs 내 아키텍처/워크플로/계획 문서 수정이 필요한지 판단하라
- 새 API, 새 폴더 구조, 새 라우트, 새 운영 방식이 생겼다면 문서 반영 후보를 제시하라

### 5. 완료 상태 판정
- Done인지 Partial Done인지 구분하라
- 완료로 보기 어려운 이유가 있으면 명확히 적어라
- 남은 리스크와 다음 단계를 정리하라

## 검증 시 반드시 봐야 하는 항목

이번 변경 기준으로 아래를 검토하라.

### 공통 검증
- lint 가능 여부
- build 가능 여부
- typecheck 가능 여부
- import 경로 정합성
- 라우트 경로 정합성
- 새 파일 참조 오류 여부

### 기능 검증
- 기존 `dev_support` 핵심 기능 영향 여부
- 신규 메뉴 진입 가능 여부
- 신규 라우트 렌더링 여부
- 신규 API 호출 흐름 확인 가능 여부
- 서버 계층 추가 시 기본 import/구조 정합성

### 보안/경계 검증
- 민감정보가 클라이언트 코드에 섞이지 않았는지
- Supabase service role 또는 외부 API key가 노출되지 않았는지
- 클라이언트/서버 경계가 무너지지 않았는지
- Discord 결합 로직을 무분별하게 이식하지 않았는지

### 문서 검증
- README 반영 필요 여부
- docs 반영 필요 여부
- 계획 문서 업데이트 필요 여부
- 변경 사항과 문서 설명 간 불일치 여부

## 최근 변경을 평가할 때 특히 확인할 것

아래 항목을 빠뜨리지 마라.

1. 이번 변경이 기존 업무 도우미 기능에 영향을 줄 가능성
2. 통합 방향과 실제 변경이 일치하는지 여부
3. 구조적으로 다음 단계의 발판이 되었는지 여부
4. 과도하게 범위가 커지지 않았는지 여부
5. 문서화 없이 넘어가면 나중에 혼선이 생길 지점
6. 아직 검증되지 않은 리스크 영역

---

## 검증 결과 작성 원칙

다음 원칙을 반드시 지켜라.

### 1. 검증한 것과 검증하지 못한 것을 구분하라
예:
- “`npm run build`는 수행했고 통과했다”
- “실제 API 연동은 환경 미비로 확인하지 못했다”

### 2. 추정과 사실을 구분하라
예:
- “현재 코드 구조상 회귀 가능성은 낮아 보인다”
- “실제 런타임까지는 미확인이다”

### 3. 완료와 부분 완료를 구분하라
- 검증이 충분하지 않으면 Done이라고 하지 마라
- 구조 골격만 있으면 Partial Done으로 표현하라

### 4. 리스크를 숨기지 마라
- import 정합성 불확실
- 환경변수 의존 미확인
- UI 진입은 가능하지만 실제 데이터 연동은 미확인
- 기존 기능 회귀는 최소 확인만 됨

이런 항목은 솔직하게 적어라
