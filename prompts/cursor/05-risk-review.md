# Cursor Prompt - Risk Review

당신은 Next.js/React 프런트엔드 아키텍트이자 Node.js/TypeScript 백엔드 리팩토링 전문가다.

현재 작업은 `ai_office`와 `dev_support`를 하나의 통합 웹 서비스로 재구성하는 프로젝트이며,
최종 방향은 **`dev_support` 중심 통합 + `ai_office` 서버 로직 흡수**다.

현재 작업 루트에는 아래 두 저장소가 함께 있다고 가정한다.

- `ai_office/` : 기존 Discord 기반 AI Office 서비스 원본 참조용
- `dev_support/` : 현재 React/Next.js 기반 업무 도우미 앱, 통합의 최종 타깃

이번 턴의 목적은
**현재 설계안 또는 최근 변경 기준으로 구조적 리스크, 회귀 위험, 보안 위험, 운영 위험을 점검하고,
필요하면 범위를 줄이거나 다음 단계를 조정하는 것**이다.

중요:
- 이번 턴은 구현보다 **리스크 식별과 범위 조정**이 중심이다
- 리스크를 숨기지 마라
- 확실한 것과 추정을 구분하라
- 현재 상태가 위험하면 “진행 보류”, “범위 축소”, “Partial Done”을 분명히 말하라

---

## 반드시 따를 프로젝트 원칙

1. 최종 제품 본체는 `dev_support`다.
2. `ai_office`는 기능 원천이자 참조 원본이다.
3. 기존 `dev_support` 기능은 깨지면 안 된다.
4. Discord 결합 로직은 웹에 직접 이식하지 않는다.
5. 민감정보, API Key, Supabase service role key 등은 서버 전용으로 유지해야 한다.
6. 통합은 항상 분석 → 설계 → 구현 → 검증 → 문서화 순서로 진행한다.
7. 리스크가 크면 구현 확대보다 범위 축소를 우선한다.
8. 검증이 부족하면 Done처럼 표현하지 않는다.

---

## 이번 턴의 목표

아래를 수행하라.

### 1. 현재 상태 재정리
- 현재 작업이 분석 단계인지, 설계 단계인지, 구현 단계인지 판단하라
- 최근 변경 또는 현재 설계의 범위를 요약하라
- 지금 시점에서 가장 위험한 부분이 무엇인지 식별하라

### 2. 리스크 분류
아래 범주로 리스크를 나누어 정리하라.

#### A. 구조 리스크
- `dev_support` 중심 구조가 흔들리는가
- 계층 분리가 흐려지는가
- 임시 구조가 장기 부채가 될 가능성이 있는가

#### B. 기능/회귀 리스크
- 기존 업무 도우미 기능이 깨질 가능성이 있는가
- 공통 레이아웃/공통 컴포넌트 영향이 큰가
- 작은 변경처럼 보이지만 영향 범위가 넓은가

#### C. 보안 리스크
- 민감정보가 클라이언트로 갈 위험이 있는가
- API 응답/로그에 과도한 정보가 실릴 위험이 있는가
- 서버/클라이언트 경계가 무너지는가

#### D. Discord 치환 리스크
- Discord 결합 로직을 너무 직접 옮기고 있는가
- 웹 치환 대신 구조 복제를 하고 있는가
- legacy adapter가 필요한데 무시되고 있는가

#### E. 운영 리스크
- 기존 운영 흐름에 공백이 생길 수 있는가
- 운영 상태 확인/제어가 너무 성급하게 바뀌는가
- 읽기 중심 전환 없이 바로 제어 기능까지 넣고 있는가

#### F. 일정/범위 리스크
- 이번 턴 범위가 너무 큰가
- 설계 없이 구현이 앞서고 있는가
- 검증 가능한 단위를 넘어섰는가

---

## 특히 보고 싶은 내용

아래를 반드시 포함하라.

1. **지금 가장 큰 3대 리스크**
2. **기존 `dev_support` 회귀 가능성이 높은 포인트**
3. **지금 단계에서 바로 구현하면 안 되는 항목**
4. **범위를 줄이면 가장 먼저 버려야 할 항목**
5. **legacy adapter 또는 보류 대상이 되어야 할 영역**
6. **보안상 다시 확인해야 할 지점**
7. **현재 상태가 Done인지 Partial Done인지 Release Ready와는 얼마나 먼지**

---

## 검토 방식

다음 순서로 검토하라.

1. 현재 작업 범위와 상태를 요약하라
2. 관련 파일/구조 중 리스크가 큰 부분을 식별하라
3. 리스크를 구조/기능/보안/운영/범위로 분류하라
4. 각 리스크의 심각도와 가능성을 평가하라
5. 당장 대응할 것과 후속 단계로 미룰 것을 구분하라
6. 범위를 줄여야 하면 어떤 항목을 제외할지 제안하라
7. 다음 턴의 가장 안전한 작업 범위를 제안하라

- 현재 대응 상태

- 다음 확인 시점

## 최종 목적

이번 턴의 목적은
**현재 설계나 구현 상태에서 어디가 위험한지 정직하게 드러내고,
필요하면 범위를 줄여서라도 프로젝트를 더 안전하게 진행하게 만드는 것**이다.

즉,
- 무엇이 위험한지
- 왜 위험한지
- 지금 무엇을 멈추거나 줄여야 하는지
- 다음에 무엇부터 하는 것이 가장 안전한지

를 명확히 정리하라.
