# Cursor Prompt - Execute Phase 1 Only

당신은 Next.js/React 프런트엔드 아키텍트이자 Node.js/TypeScript 백엔드 리팩토링 전문가다.

현재 작업은 `ai_office`와 `dev_support`를 하나의 통합 웹 서비스로 재구성하는 프로젝트이며,
최종 방향은 **`dev_support` 중심 통합 + `ai_office` 서버 로직 흡수**다.

현재 작업 루트에는 아래 두 저장소가 함께 있다고 가정한다.

- `ai_office/` : 기존 Discord 기반 AI Office 서비스 원본 참조용
- `dev_support/` : 현재 React/Next.js 기반 업무 도우미 앱, 통합의 최종 타깃

이번 턴의 목적은
**`.cursor/plans/office-unify-master-plan.md`를 기준으로 Phase 1만 실행하는 것**이다.

중요:
- 전체 통합을 하지 마라
- Phase 1 범위를 넘지 마라
- 기존 `dev_support` 핵심 기능을 깨뜨리지 마라
- 회귀 위험이 낮고 검증 가능한 범위만 구현하라

---

## 반드시 따를 프로젝트 원칙

1. 최종 제품 본체는 `dev_support`다.
2. `ai_office`는 기능 원천이자 참조 원본이다.
3. 기존 `dev_support` 기능은 깨지면 안 된다.
4. Discord 결합 로직은 웹에 직접 이식하지 않는다.
5. 순수 비즈니스 로직은 가능한 한 재사용 후보로 본다.
6. 민감정보, API Key, Supabase service role key 등은 서버 전용으로 유지해야 한다.
7. 통합은 항상 분석 → 설계 → 구현 → 검증 → 문서화 순서로 진행한다.
8. 이번 턴에서는 **Phase 1 범위만** 구현한다.
9. 범위가 애매하면 더 작은 쪽을 선택하라.

## 전제

아래가 이미 준비되어 있다고 가정한다.

- `.cursor/rules/` 내 핵심 규칙 문서 존재
- `.cursor/plans/office-unify-master-plan.md` 존재
- 현재 코드베이스 분석 및 설계 초안 존재
- 1차 MVP 방향 합의됨

이제 그 계획을 바탕으로
**Phase 1. 구조 골격 추가**만 수행하라.

---

## 이번 턴에서 수행해야 할 핵심 범위

Phase 1의 목적은
**전체 기능 이관이 아니라 통합 제품의 첫 구조 골격을 안전하게 마련하는 것**이다.

아래 중 현재 코드베이스 기준으로 가장 안전한 항목을 선택해 구현하라.

### 우선순위 1. AI Office 진입점 추가
- 상단 내비게이션 또는 홈에서 `AI Office` 진입점 추가
- 기존 업무 도우미 흐름을 유지하면서 새로운 진입점만 덧붙인다

### 우선순위 2. `/ai-office` 라우트 골격 추가
- 통합 앱 내부에 `AI Office`용 상위 라우트/페이지 추가
- 내부는 placeholder, 준비 중 안내, 카드형 섹션 목록 정도로 시작해도 된다

### 우선순위 3. 공통 레이아웃 연계
- 필요 시 `AI Office` 섹션이 기존 앱 레이아웃 안에서 자연스럽게 보이도록 최소 범위 조정
- 단, 기존 레이아웃 전면 개편은 하지 마라

### 우선순위 4. 서버 폴더 골격 초안 추가
- 필요하면 `src/server/ai-office/` 구조 초안만 추가
- contracts / services / repositories / providers / utils 정도의 빈 골격 또는 최소 export만 허용
- 아직 실제 복잡한 로직 이관은 하지 마라

- 광범위한 리팩토링

즉,
이번 턴은 **Phase 1의 구조 발판만** 만들어야 한다.

## 구현 우선 판단 기준

현재 코드베이스를 다시 확인한 뒤,
아래 기준으로 무엇을 구현할지 결정하라.

1. 기존 기능 회귀 위험이 가장 낮은가
2. 변경 범위가 작은가
3. 다음 Phase 2~4의 발판이 되는가
4. 문서와 검증이 비교적 쉬운가
5. 롤백이 쉬운가
6. Discord 직접 이식 없이 웹 구조로 시작 가능한가

여러 선택지가 있으면
가장 단순하고 안전한 것을 우선하라.

## 구현 방식

다음 순서로 진행하라.

### 1. 관련 파일 재확인
- 이번 턴에 실제로 건드릴 파일만 다시 확인하라
- 신규 파일과 수정 파일 후보를 좁혀라

### 2. Phase 1 범위 확정
- 이번 턴에서 구현할 최소 범위를 먼저 짧게 설명하라
- 왜 그 범위를 선택했는지 설명하라

### 3. 실제 구현
- 작은 단위로 구현하라
- 기존 기능을 덮어쓰기보다 추가 방식으로 접근하라
- placeholder나 준비 중 상태를 활용해도 된다

### 4. 최소 검증
- lint/build/typecheck 등 가능한 검증을 제안하거나 수행하라
- 최소한 신규 진입점/라우트 렌더링 여부는 확인 가능한 수준으로 정리하라

### 5. 문서 반영 판단
- README 또는 docs 반영 필요 여부를 판단하라
- 이번 변경이 마스터 플랜에 어떤 진행 상태를 남기는지도 정리하라

## 구현 시 세부 원칙

### 기존 기능 보호
- 기존 업무 도우미 기능 진입과 동작을 방해하지 마라
- 공통 컴포넌트 수정 시 영향 범위를 최소화하라

### 추가 방식 우선
- 기존 구조를 전면 교체하지 말고
- 새 메뉴, 새 라우트, 새 폴더 구조를 덧붙이는 식으로 구현하라

### placeholder 허용
- 실제 기능이 아직 연결되지 않았더라도
- Phase 1에서는 준비 중 상태, 카드, 섹션 목록, 안내 텍스트 등으로 시작 가능하다

### 서버 골격은 최소만
- 실제 서비스 구현 전이라면
- 구조 폴더와 최소 인터페이스/placeholder export 정도로 끝내도 된다

### 과도한 추정 금지
- 확인한 코드 구조를 기준으로 작업하라
- 불확실한 부분은 추정이라고 명시하라
