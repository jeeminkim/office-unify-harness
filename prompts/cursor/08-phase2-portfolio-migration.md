# Cursor Prompt - Execute Phase 2 Portfolio Migration Only

당신은 Next.js/React 프런트엔드 아키텍트이자 Node.js/TypeScript 백엔드 리팩토링 전문가다.

현재 작업은 `ai_office`와 `dev_support`를 하나의 통합 웹 서비스로 재구성하는 프로젝트이며,
최종 방향은 **`dev_support` 중심 통합 + `ai_office` 서버 로직 흡수**다.

현재 작업 루트에는 아래 두 저장소가 함께 있다고 가정한다.

- `ai_office/` : 기존 Discord 기반 AI Office 서비스 원본 참조용
- `dev_support/` : 현재 React/Next.js 기반 업무 도우미 앱, 통합의 최종 타깃

이번 턴의 목적은
**`.cursor/plans/office-unify-master-plan.md`를 기준으로 Phase 2인 포트폴리오 조회 기능만 안전하게 이관하는 것**이다.

중요:
- 전체 통합을 하지 마라
- Phase 2 범위를 넘지 마라
- 기존 `dev_support` 핵심 기능을 깨뜨리지 마라
- 이번 단계에서는 **읽기 중심 포트폴리오 조회 흐름**만 우선한다
- 매수/매도/수정/삭제 같은 쓰기 기능은 필요하지 않으면 넣지 마라

---

## 반드시 따를 프로젝트 원칙

1. 최종 제품 본체는 `dev_support`다.
2. `ai_office`는 기능 원천이자 참조 원본이다.
3. 기존 `dev_support` 기능은 깨지면 안 된다.
4. Discord 결합 로직은 웹에 직접 이식하지 않는다.
5. 순수 비즈니스 로직은 가능한 한 재사용 후보로 본다.
6. 민감정보, API Key, Supabase service role key 등은 서버 전용으로 유지해야 한다.
7. 통합은 항상 분석 → 설계 → 구현 → 검증 → 문서화 순서로 진행한다.
8. 이번 턴에서는 **Phase 2 범위만** 구현한다.
9. 포트폴리오 기능은 우선 **조회 중심, 읽기 중심, 회귀 위험 낮은 범위**로 구현한다.

## 이번 턴의 핵심 목표

이번 턴의 목표는
**`ai_office`의 포트폴리오 관련 로직 중 웹으로 옮기기 적합한 읽기 중심 흐름을 `dev_support` 내부의 서버/웹 구조로 안전하게 이관하는 것**이다.

즉, 아래를 우선한다.

1. 포트폴리오 조회 관련 순수 로직 식별
2. 서버 전용 데이터 접근 구조 정리
3. 읽기 전용 API 또는 서버 진입점 추가
4. `/ai-office/portfolio` 또는 이에 준하는 UI 연결
5. 최소한의 조회 결과 표시
6. 기존 기능 회귀 방지
7. 검증 가능한 최소 범위 유지

---

## 우선 구현 대상

현재 코드베이스 기준으로 아래 중 가장 안전한 범위를 선택하라.

### 우선순위 1. 서버 조회 구조 추가
- 포트폴리오 조회용 service / repository / contract 골격 또는 실제 최소 구현 추가
- Supabase 또는 기존 데이터 소스 접근은 서버 전용으로 유지
- 민감정보와 외부 의존은 클라이언트로 보내지 마라

### 우선순위 2. 읽기 전용 API 1개 추가
- 예: `GET /api/ai-office/portfolio`
- 목적은 포트폴리오 데이터 조회
- 입력이 필요하면 최소화하라
- 응답은 UI에 필요한 최소 구조로 제한하라

### 우선순위 3. 포트폴리오 페이지 연결
- 예: `/ai-office/portfolio`
- 초기에는 단순 목록/요약/상태 카드로 시작 가능
- 데이터가 없을 때 빈 상태를 자연스럽게 보여줘라

### 우선순위 4. 최소 DTO / 타입 구조 정리
- portfolio item type
- portfolio summary type
- response contract
- UI에서 과도한 내부 구조를 직접 알지 않게 구성하라

## 구현 우선 판단 기준

현재 코드베이스를 다시 확인한 뒤,
아래 기준으로 구현 범위를 결정하라.

1. 읽기 전용으로 먼저 완성 가능한가
2. 서버/클라이언트 경계가 명확한가
3. 기존 `dev_support` 기능 회귀 위험이 낮은가
4. 포트폴리오 데이터 흐름을 다음 Phase의 기반으로 쓸 수 있는가
5. API/서비스/페이지 연결이 과도하게 복잡하지 않은가
6. 검증 가능한 최소 단위인가

여러 선택지가 있으면
가장 작은 범위, 가장 명확한 조회 흐름을 우선하라.

## 구현 방식

다음 순서로 진행하라.

### 1. 관련 파일 재확인
- `ai_office`에서 포트폴리오 조회 관련 파일/서비스/유틸을 식별하라
- `dev_support`에서 Phase 1 구조와 연결 가능한 파일을 식별하라
- 수정 파일과 참조 파일을 구분하라

### 2. 읽기 중심 이관 범위 확정
- 이번 턴에서 옮길 포트폴리오 조회 범위를 먼저 짧게 설명하라
- 왜 그 범위가 가장 안전한지 설명하라

### 3. 서버 구조 우선 구현
- 가능하면 server/service/repository/contract 구조를 먼저 세워라
- 데이터 접근은 서버에 둬라
- 클라이언트는 응답을 소비하는 쪽으로 두어라

### 4. 최소 UI 연결
- `/ai-office/portfolio` 페이지 또는 기존 `AI Office` 섹션 안에서 연결하라
- 데이터 표시가 단순해도 괜찮다
- 로딩/빈 상태/오류 상태를 최소한 고려하라

### 5. 최소 검증
- lint/build/typecheck 등 가능한 검증을 제안하거나 수행하라
- 최소한 라우트 진입, 조회 흐름, 빈 상태 또는 기본 응답 구조를 설명하라

### 6. 문서 반영 판단
- README 또는 docs 반영 필요 여부를 판단하라
- master plan의 Phase 2 진행 상태 반영 필요 여부도 정리하라

## 구현 시 세부 원칙

### 읽기 중심 우선
- 이번 단계에서는 쓰기 기능보다 조회 기능을 우선한다
- 사용자가 포트폴리오 상태를 보는 최소 흐름을 먼저 만든다

### 순수 로직 우선 재사용
- 데이터 가공, 요약, 정렬, 포맷팅 로직 중 재사용 가능한 부분을 우선 검토하라
- Discord 표시용 포맷 로직은 그대로 옮기지 마라

### 서버 전용 경계 유지
- Supabase 접근, API key, service role, 외부 조회는 서버 전용으로 둔다
- 클라이언트는 최소 응답만 받도록 하라

### UI는 단순해도 괜찮다
- 완성형 대시보드가 아니라
- 목록, 요약 카드, 상태 문구 정도로 시작해도 된다

### 과도한 데이터 노출 금지
- 내부 구조 전체를 응답으로 넘기지 마라
- UI에 필요한 최소 필드만 전달하라
