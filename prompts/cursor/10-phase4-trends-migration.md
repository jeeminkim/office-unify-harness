# Cursor Prompt - Execute Phase 4 Trends Migration Only

당신은 Next.js/React 프런트엔드 아키텍트이자 Node.js/TypeScript 백엔드 리팩토링 전문가다.

현재 작업은 `ai_office`와 `dev_support`를 하나의 통합 웹 서비스로 재구성하는 프로젝트이며,
최종 방향은 **`dev_support` 중심 통합 + `ai_office` 서버 로직 흡수**다.

현재 작업 루트에는 아래 두 저장소가 함께 있다고 가정한다.

- `ai_office/` : 기존 Discord 기반 AI Office 서비스 원본 참조용
- `dev_support/` : 현재 React/Next.js 기반 업무 도우미 앱, 통합의 최종 타깃

이번 턴의 목적은
**`.cursor/plans/office-unify-master-plan.md`를 기준으로 Phase 4인 트렌드 분석 실행 흐름만 안전하게 이관하는 것**이다.

중요:
- 전체 통합을 하지 마라
- Phase 4 범위를 넘지 마라
- 기존 `dev_support` 핵심 기능을 깨뜨리지 마라
- 이번 단계에서는 **트렌드 분석의 최소 웹 실행 흐름**만 우선한다
- Discord interaction, panel restore, ephemeral reply 구조를 그대로 옮기지 마라
- 고급 운영 제어, 대규모 소스 통합, 모든 트렌드 카테고리 완전 구현은 필요하면 후속 단계로 미뤄라

---

## 반드시 따를 프로젝트 원칙

1. 최종 제품 본체는 `dev_support`다.
2. `ai_office`는 기능 원천이자 참조 원본이다.
3. 기존 `dev_support` 기능은 깨지면 안 된다.
4. Discord 결합 로직은 웹에 직접 이식하지 않는다.
5. 순수 비즈니스 로직은 가능한 한 재사용 후보로 본다.
6. 민감정보, API Key, Supabase service role key 등은 서버 전용으로 유지해야 한다.
7. 통합은 항상 분석 → 설계 → 구현 → 검증 → 문서화 순서로 진행한다.
8. 이번 턴에서는 **Phase 4 범위만** 구현한다.
9. 트렌드 기능은 우선 **실행 중심, 읽기 중심 결과 표시, 회귀 위험 낮은 범위**로 구현한다.

## 이번 턴의 핵심 목표

이번 턴의 목표는
**`ai_office`의 트렌드 분석 관련 로직 중 웹으로 옮기기 적합한 최소 실행 흐름을 `dev_support` 내부의 서버/웹 구조로 안전하게 이관하는 것**이다.

즉, 아래를 우선한다.

1. 트렌드 분석 관련 순수 로직 식별
2. 서버 전용 트렌드 실행 service/provider/contract 구조 정리
3. 최소 실행 API 또는 서버 진입점 추가
4. `/ai-office/trends` 또는 이에 준하는 UI 연결
5. 사용자가 주제/카테고리/질문을 넣고 실행하는 최소 흐름 마련
6. 결과를 웹 패널에 읽기 좋게 표시
7. 기존 기능 회귀 방지
8. 검증 가능한 최소 범위 유지

## 우선 구현 대상

현재 코드베이스 기준으로 아래 중 가장 안전한 범위를 선택하라.

### 우선순위 1. 서버 트렌드 실행 구조 추가
- trends service / provider / contract 골격 또는 최소 구현 추가
- LLM 호출은 서버 전용으로 유지
- provider 선택이나 모델 호출 로직은 클라이언트로 보내지 마라

### 우선순위 2. 트렌드 실행 API 1개 추가
- 예: `POST /api/ai-office/trends`
- 입력은 최소화하라
  - 예: 주제, 카테고리, 질문, 모드
- 응답은 UI 표시용 최소 구조로 제한하라
  - 예: 요약, 핵심 포인트, 카테고리, 결론, 상태

### 우선순위 3. 트렌드 페이지 연결
- 예: `/ai-office/trends`
- 최소 입력 폼 + 실행 버튼 + 결과 패널 구조로 시작 가능
- 아직 고급 필터, 다중 소스 비교, 장기 저장, 상세 히스토리까지 넣지 않아도 된다

### 우선순위 4. 결과 DTO / 타입 구조 정리
- trends request type
- trends response type
- trend panel item / summary type
- UI가 내부 provider 구현을 직접 알지 않게 구성하라

- 실시간 크롤링/대규모 외부 소스 통합을 한 번에 시도
- AI 토론/포트폴리오 기능 동시 구현
- 전체 LLM provider 통합 체계 완성 시도
- DB 스키마 대규모 변경
- 인증/권한 구조 전면 도입
- 결과 UI의 전면 완성형 대시보드 구현

즉,
이번 턴은 **트렌드 분석의 최소 웹 실행 버전**만 다뤄야 한다.

## 구현 우선 판단 기준

현재 코드베이스를 다시 확인한 뒤,
아래 기준으로 구현 범위를 결정하라.

1. 서버에서 안전하게 실행 가능한가
2. Discord 결합 구조 없이 웹 요청/응답으로 재구성 가능한가
3. 결과를 최소 UI로 표시할 수 있는가
4. 기존 `dev_support` 기능 회귀 위험이 낮은가
5. 이후 운영 상태/확장 기능 전에 공통 실행 패턴 기반이 되는가
6. 검증 가능한 최소 단위인가

여러 선택지가 있으면
가장 작은 범위, 가장 명확한 실행/응답 흐름을 우선하라.

## 구현 방식

다음 순서로 진행하라.

### 1. 관련 파일 재확인
- `ai_office`에서 트렌드 분석 관련 파일/서비스/유틸을 식별하라
- provider, service, 카테고리 처리, 결과 요약 관련 영역을 확인하라
- `dev_support`에서 Phase 1~3 구조와 연결 가능한 파일을 식별하라
- 수정 파일과 참조 파일을 구분하라

### 2. 최소 트렌드 흐름 확정

- 왜 그 범위가 가장 안전한지 설명하라
- 어떤 고급 기능은 의도적으로 제외하는지 적어라

### 3. 서버 구조 우선 구현
- trends service / provider / contract / API 구조를 먼저 세워라
- LLM 호출 및 민감정보는 서버에 둬라
- 클라이언트는 최소 입력과 응답 소비 역할로 제한하라

### 4. 최소 UI 연결
- `/ai-office/trends` 페이지 또는 기존 `AI Office` 섹션 안에서 연결하라
- 입력 폼, 실행 버튼, 로딩 상태, 결과 패널을 최소 수준으로 마련하라
- 결과가 없을 때 빈 상태를 자연스럽게 보여줘라

### 5. 최소 검증
- lint/build/typecheck 등 가능한 검증을 제안하거나 수행하라
- 최소한 라우트 진입, 폼 제출 흐름, 응답 구조, 결과 렌더링 여부를 설명하라

### 6. 문서 반영 판단
- README 또는 docs 반영 필요 여부를 판단하라
- master plan의 Phase 4 진행 상태 반영 필요 여부도 정리하라

## 구현 시 세부 원칙

### Discord 구조 복제 금지
- Discord interaction, panel restore, ephemeral 결과 모델을 그대로 구현하지 마라
- 웹에서는 폼 제출 → API 요청 → 결과 패널 표시 흐름으로 재구성하라

### 실행 흐름 최소화
- 복잡한 옵션보다 기본 실행 흐름을 먼저 만든다

- provider 키, 모델 호출, 민감한 처리 로직은 서버 전용으로 둔다
- 클라이언트는 결과 소비자여야 한다

- 완성형 트렌드 대시보드보다
- 요약, 핵심 포인트, 카테고리, 결론, 상태 메시지 위주로 시작해도 된다

### 과도한 데이터 노출 금지
- 내부 디버깅 정보, provider 세부 정보, 민감한 로직 구조를 UI 응답에 과도하게 노출하지 마라

### 추정과 확인 구분
- 현재 코드베이스에서 확인한 것과 추정한 것을 구분하라
- 미검증 항목은 숨기지 마라
