# Cursor Prompt - Execute Phase 5 Ops Read-Only Only

당신은 Next.js/React 프런트엔드 아키텍트이자 Node.js/TypeScript 백엔드 리팩토링 전문가다.

현재 작업은 `ai_office`와 `dev_support`를 하나의 통합 웹 서비스로 재구성하는 프로젝트이며,
최종 방향은 **`dev_support` 중심 통합 + `ai_office` 서버 로직 흡수**다.

현재 작업 루트에는 아래 두 저장소가 함께 있다고 가정한다.

- `ai_office/` : 기존 Discord 기반 AI Office 서비스 원본 참조용
- `dev_support/` : 현재 React/Next.js 기반 업무 도우미 앱, 통합의 최종 타깃

이번 턴의 목적은
**`.cursor/plans/office-unify-master-plan.md`를 기준으로 Phase 5인 운영 상태 페이지를 읽기 전용으로만 안전하게 이관하는 것**이다.

중요:
- 전체 통합을 하지 마라
- Phase 5 범위를 넘지 마라
- 기존 `dev_support` 핵심 기능을 깨뜨리지 마라
- 이번 단계에서는 **운영/제어 기능이 아니라 읽기 중심 상태 페이지**만 우선한다
- 재기동, 강제 실행, 관리자 액션 같은 제어 기능은 필요하면 후속 단계로 미뤄라
- Discord 운영 흐름을 웹에서 그대로 복제하지 마라

---

## 반드시 따를 프로젝트 원칙

1. 최종 제품 본체는 `dev_support`다.
2. `ai_office`는 기능 원천이자 참조 원본이다.
3. 기존 `dev_support` 기능은 깨지면 안 된다.
4. Discord 결합 로직은 웹에 직접 이식하지 않는다.
5. 순수 비즈니스 로직은 가능한 한 재사용 후보로 본다.
6. 민감정보, API Key, Supabase service role key 등은 서버 전용으로 유지해야 한다.
7. 통합은 항상 분석 → 설계 → 구현 → 검증 → 문서화 순서로 진행한다.
8. 이번 턴에서는 **Phase 5 범위만** 구현한다.
9. 운영 기능은 우선 **상태 확인, 읽기 전용, 회귀 위험 낮은 범위**로 구현한다.

## 이번 턴의 핵심 목표

이번 턴의 목표는
**`ai_office`의 운영/상태 관련 정보 중 웹으로 옮기기 적합한 읽기 중심 흐름을 `dev_support` 내부의 서버/웹 구조로 안전하게 이관하는 것**이다.

즉, 아래를 우선한다.

1. 운영 상태 관련 순수 조회 로직 식별
2. 서버 전용 상태 조회 service / contract / API 구조 정리
3. 최소 상태 조회 API 또는 서버 진입점 추가
4. `/ai-office/ops` 또는 이에 준하는 UI 연결
5. 사용자 또는 운영자가 읽기 좋게 상태를 확인할 수 있는 최소 패널 마련
6. 기존 기능 회귀 방지
7. 검증 가능한 최소 범위 유지

## 우선 구현 대상

현재 코드베이스 기준으로 아래 중 가장 안전한 범위를 선택하라.

### 우선순위 1. 서버 상태 조회 구조 추가
- ops status service / repository / contract 골격 또는 최소 구현 추가
- 운영 상태 판단이나 정보 수집은 서버 전용으로 유지
- 민감한 내부 제어 로직은 클라이언트에 보내지 마라

### 우선순위 2. 읽기 전용 상태 API 1개 추가
- 예: `GET /api/ai-office/ops/status`
- 응답은 최소화하라
  - 예: 서비스 상태, 마지막 업데이트 시각, 사용 가능 기능, 경고 메시지, 기본 헬스 상태
- 내부 제어 명령, 비밀 정보, 불필요한 운영 세부사항은 응답에 넣지 마라

### 우선순위 3. 운영 상태 페이지 연결
- 예: `/ai-office/ops`
- 카드, 상태 배지, 최근 상태 메시지, 준비 중 기능 안내 정도로 시작 가능
- 아직 버튼 기반 제어 기능까지 넣지 않아도 된다

### 우선순위 4. 상태 DTO / 타입 구조 정리
- ops status response type
- status card item type
- warning / health summary type
- UI가 내부 운영 구현 세부사항을 직접 알지 않게 구성하라

## 구현 우선 판단 기준

현재 코드베이스를 다시 확인한 뒤,
아래 기준으로 구현 범위를 결정하라.

1. 읽기 전용으로 안전하게 제공 가능한가
2. 제어 기능 없이도 의미 있는 상태 정보를 보여줄 수 있는가
3. 민감한 운영 세부사항을 과도하게 노출하지 않는가
4. 기존 `dev_support` 기능 회귀 위험이 낮은가
5. 이후 인증/운영 제어 고도화 전에 상태 가시화 기반이 되는가
6. 검증 가능한 최소 단위인가

여러 선택지가 있으면
가장 작은 범위, 가장 명확한 상태 조회 흐름을 우선하라.

- 최소한 라우트 진입, 상태 응답 구조, UI 렌더링 여부를 설명하라

### 6. 문서 반영 판단

## 최종 목적

이번 턴의 목적은
**통합 제품 안에서 운영 상태 확인 흐름의 첫 번째 웹 전환 버전을 안전하게 만드는 것**이다.

즉,
- 서버 전용 상태 조회 구조를 유지하고
- 웹에서는 최소 읽기 전용 상태 표시만 제공하며
- 기존 기능을 깨지 않고
- 이후 인증/운영 제어 고도화로 확장 가능한 기반을 만들라.

출력은 한국어로 작성하라.
