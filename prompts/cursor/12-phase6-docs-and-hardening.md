# Cursor Prompt - Execute Phase 6 Docs and Hardening Only

당신은 Next.js/React 프런트엔드 아키텍트이자 Node.js/TypeScript 백엔드 리팩토링 전문가다.

현재 작업은 `ai_office`와 `dev_support`를 하나의 통합 웹 서비스로 재구성하는 프로젝트이며,
최종 방향은 **`dev_support` 중심 통합 + `ai_office` 서버 로직 흡수**다.

현재 작업 루트에는 아래 두 저장소가 함께 있다고 가정한다.

- `ai_office/` : 기존 Discord 기반 AI Office 서비스 원본 참조용
- `dev_support/` : 현재 React/Next.js 기반 업무 도우미 앱, 통합의 최종 타깃

이번 턴의 목적은
**`.cursor/plans/office-unify-master-plan.md`를 기준으로 Phase 6인 문서 정리와 구조 안정화(hardening)만 수행하는 것**이다.

중요:
- 이번 턴은 새로운 대형 기능 추가가 목적이 아니다
- Phase 6 범위를 넘지 마라
- 기존 `dev_support` 핵심 기능을 깨뜨리지 마라
- 지금까지 반영된 구조와 기능을 기준으로 검증, 문서화, 정리, 안정화에 집중하라
- 구현이 필요하더라도 작은 보정 수준만 허용한다
- 새 기능 욕심보다 품질 정리와 릴리즈 준비도를 우선하라

---

## 반드시 따를 프로젝트 원칙

1. 최종 제품 본체는 `dev_support`다.
2. `ai_office`는 기능 원천이자 참조 원본이다.
3. 기존 `dev_support` 기능은 깨지면 안 된다.
4. Discord 결합 로직은 웹에 직접 이식하지 않는다.
5. 민감정보, API Key, Supabase service role key 등은 서버 전용으로 유지해야 한다.
6. 통합은 항상 분석 → 설계 → 구현 → 검증 → 문서화 순서로 진행한다.
7. 이번 턴에서는 **Phase 6 범위만** 수행한다.
8. 이번 단계의 목표는 기능 확장보다 **검증 보강, 문서 갱신, 구조 안정화**다.
9. 지금 단계에서 큰 리팩토링이 필요해 보여도, 우선은 문서화와 위험 축소 중심으로 접근하라.

## 전제

아래가 이미 어느 정도 준비되어 있다고 가정한다.

- `.cursor/rules/` 내 핵심 규칙 문서 존재
- `.cursor/plans/office-unify-master-plan.md` 존재
- Phase 1~5 중 일부 또는 전부가 반영됨
- `AI Office` 진입 구조, 일부 페이지, 일부 API, 일부 서버 골격이 존재할 수 있음
- 현재 코드베이스 분석 및 이전 작업 결과가 존재함

이제 그 상태를 바탕으로
**Phase 6. 문서화 및 안정화**만 수행하라.

## 이번 턴의 핵심 목표

이번 턴의 목표는
**지금까지 반영된 통합 구조와 기능을 기준으로, 문서·검증·리스크·구조 정합성을 정리하여 다음 단계 또는 릴리즈 판단이 가능하도록 만드는 것**이다.

즉, 아래를 우선한다.

1. 현재 반영된 구조/기능 상태 재정리
2. README 및 docs 문서 갱신
3. master plan 진행 상태 반영
4. 검증 가능 항목 정리 및 수행
5. 미검증/불안정/보류 항목 분리
6. 사소하지만 필요한 구조 보정
7. 릴리즈 전 상태 판단이 가능하도록 정리

## 우선 수행 대상

현재 코드베이스 기준으로 아래 중 필요한 범위를 수행하라.

### 우선순위 1. 현재 반영 상태 정리
- Phase 1~5 중 무엇이 실제로 반영되었는지 정리
- 현재 존재하는 라우트, API, 서버 구조, UI 진입점 정리
- 구현됨 / 부분 구현 / 미구현 상태를 구분

### 우선순위 2. README 갱신
- 현재 앱 구조 설명
- AI Office 진입 구조 설명
- 실행/검증 방법 갱신
- 새 라우트 또는 기능 상태 설명
- 아직 미완성인 부분은 솔직하게 표시

### 우선순위 3. docs 갱신
- 마스터 플랜 진행 상태 반영
- 아키텍처 문서와 실제 구조 간 차이 조정
- 체크리스트/가이드 문서와 실제 운영 상태 정합성 점검
- 필요한 경우 리스크 로그 반영

### 우선순위 4. 검증 보강
- lint/build/typecheck 등 가능한 검증 수행 또는 정리
- 새 라우트/API/UI 흐름 최소 점검
- 기존 `dev_support` 핵심 기능 영향 여부 재점검
- Done / Partial Done / Release Ready 여부 판단

### 우선순위 5. 구조 hardening
- import 경로 오류, naming 불일치, 폴더 정합성, placeholder 정리 등
- 작은 범위의 안정화만 허용
- 대규모 리팩토링은 하지 마라

## 이번 턴에서 하지 말아야 할 것

아래는 이번 턴에서 금지한다.

- Phase 2~5에 해당하는 신규 기능 대규모 확장

- Discord 운영 기능의 제어 기능 이식
- 인증/권한 체계 전면 도입
- DB 스키마 대규모 개편
- 전체 라우팅/레이아웃 전면 재설계
- 공통 구조 전면 리팩토링
- 문서화 단계라는 명분으로 범위를 몰래 확장하는 것

즉,
이번 턴은 **문서, 검증, 안정화, 상태 정리**에 집중해야 한다.

## 구현/정리 판단 기준

현재 코드베이스를 다시 확인한 뒤,
아래 기준으로 Phase 6 범위를 결정하라.

1. 지금 상태를 설명 가능한가
2. 문서와 실제 구조가 어긋나지 않는가
3. 검증 가능한 범위를 더 명확히 만들 수 있는가
4. 기존 기능 회귀 위험을 더 줄일 수 있는가
5. 작은 보정으로 안정성이 올라가는가
6. 릴리즈 또는 다음 단계 판단에 도움이 되는가

여러 선택지가 있으면
기능 추가보다 **상태 정리와 검증 가능성 향상**을 우선하라.

## 수행 방식

다음 순서로 진행하라.

### 1. 현재 상태 재스캔
- 현재 존재하는 주요 라우트, API, 서버 폴더, UI 진입점을 확인하라
- 실제 반영된 것과 설계만 존재하는 것을 구분하라
- README/docs와 실제 상태가 다른 부분을 찾하라

### 2. 문서 반영 범위 확정
- 이번 턴에서 어떤 문서를 갱신할지 먼저 짧게 설명하라
- 왜 그 문서들이 필요한지 설명하라

### 3. 검증 항목 정리 및 수행
- lint/build/typecheck 등 가능한 검증을 제안하거나 수행하라
- 신규 기능과 기존 기능에 대한 최소 확인 범위를 정리하라
- 미검증 항목은 별도로 적어라

### 4. 최소 hardening 수행
- import 경로 보정, placeholder 문구 보완, naming 정리, 구조 정합성 보완 등
- 작은 안정화 작업만 수행하라
- 대규모 구조 변경은 하지 마라

### 5. 상태 판정 및 다음 단계 정리
- 현재가 Done인지 Partial Done인지 판단하라
- Release Ready 여부도 가능하면 함께 판단하라
- 남은 리스크와 다음 단계 제안을 정리하라

## 세부 원칙

### 문서와 실제 상태 정합성 우선
- README, docs, plans가 실제 코드 상태와 어긋나면 갱신하라
- “계획상 있음”과 “실제로 구현됨”을 구분하라

### 검증한 것과 못한 것 구분
- 실행한 검증은 명확히 적어라
- 환경 미비로 못한 것은 숨기지 마라

### 작은 보정만 허용
- hardening은 작은 안정화 작업만 의미한다
- 기능 확장이나 대규모 리팩토링은 금지다

### 기존 기능 보호 재점검
- `dev_support` 기존 핵심 기능 영향 여부를 다시 보라
- 통합 구조가 기존 사용자 흐름을 해치지 않는지 확인하라

### Release Ready와 Done 구분
- 구현 일부가 끝났다고 바로 릴리즈 가능으로 보지 마라
- Done / Partial Done / Partial Release / Release Ready를 구분하라
