# Office Unify Risk Log Template

## 문서 목적
이 문서는 `office-unify` 통합 프로젝트에서 발생하는
구조적 리스크, 기능 회귀 위험, 보안 이슈, 미확인 사항을
일관된 형식으로 기록하고 관리하기 위한 템플릿이다.

이 템플릿의 목적은 다음과 같다.

- 현재 리스크를 숨기지 않고 명시적으로 기록
- 구현 전/중/후의 불확실성 추적
- 회귀 위험 조기 식별
- 구조적/보안적 문제 누락 방지
- 다음 단계 의사결정의 근거 확보

이 문서는 설계 단계, 구현 단계, 검증 단계 모두에서 사용할 수 있다.

# 2. 리스크 관리 원칙

이 문서를 사용할 때는 아래 원칙을 따른다.

1. 리스크는 추정이라도 숨기지 않고 적는다.
2. 확인된 사실과 추정 위험을 구분한다.
3. 리스크는 가능한 한 영향 범위와 함께 적는다.
4. 대응 방안이 미정이어도 우선 기록한다.
5. 리스크가 해소되면 상태를 갱신한다.
6. 회귀 위험과 보안 이슈는 별도로 명확히 적는다.
7. Done 여부 판단 시 이 문서의 내용을 참고한다.

# 3. 리스크 상태 정의

## Open
- 현재 존재하며 아직 해결되지 않은 리스크

## Monitoring
- 즉시 문제는 아니지만 계속 관찰이 필요한 리스크

## Mitigated
- 완전히 제거되지는 않았지만 대응책이 적용된 리스크

## Resolved
- 현재 기준으로 해소되었고 추가 조치가 필요하지 않은 리스크

## Deferred
- 현재 단계에서는 해결하지 않고 후속 단계로 넘긴 리스크

# 6. 리스크 분류별 체크 포인트

## 6.1 구조 리스크
아래 항목을 점검한다.

- `dev_support` 중심 구조 원칙이 깨지고 있지 않은가
- `ai_office` 로직이 무분별하게 혼입되고 있지 않은가
- UI/서버/DB/Provider 경계가 흐려지고 있지 않은가
- 다음 단계 확장을 어렵게 만드는 임시 구조가 늘어나고 있지 않은가

---

## 6.2 기능/회귀 리스크
아래 항목을 점검한다.

- 기존 업무 도우미 핵심 기능이 영향을 받을 가능성
- 신규 메뉴/라우트 추가가 기존 진입 흐름을 깨뜨릴 가능성
- 작은 변경처럼 보이지만 공통 컴포넌트에 넓게 영향을 주는지 여부
- 기존 API 또는 상태 흐름과 충돌 가능성

---

## 6.3 보안 리스크
아래 항목을 점검한다.

- API Key 또는 service role key 노출 가능성
- 클라이언트 코드에 서버 전용 로직이 섞이지 않았는가
- 응답 데이터에 과도한 정보가 포함되지 않았는가
- 로그에 민감정보가 남지 않는가

---

## 6.4 운영 리스크
아래 항목을 점검한다.

- 기존 Discord 운영 흐름 제거 시 운영 공백이 생기지 않는가
- legacy adapter 없이 급격히 전환해도 되는가
- 운영 상태 페이지가 실제 운영 요구를 충분히 반영하는가
- 재기동/알림/상태 표시 흐름이 단절되지 않는가

---

## 6.5 데이터 리스크
아래 항목을 점검한다.

- source of truth가 흐려지지 않는가
- localStorage 의존이 과도해지지 않는가
- 서버 상태와 클라이언트 상태가 충돌하지 않는가
- 기존 데이터 구조를 바꾸는 경우 회귀 위험이 있는가

---

## 6.6 문서 리스크
아래 항목을 점검한다.

- 실제 구조 변경이 문서에 반영되지 않았는가
- README와 실제 실행 방식이 달라지지 않았는가
- 계획 문서와 실제 구현 상태가 어긋나지 않았는가
- 다음 작업자가 이어받기 어려운 상태가 아닌가

# 10. 사용 원칙

이 문서는 다음처럼 활용한다.

- 분석 단계: 잠재 리스크 초기 기록
- 설계 단계: 구조적/보안적 리스크 구체화
- 구현 단계: 회귀 위험 및 범위 확대 리스크 기록
- 검증 단계: 미확인 항목과 실제 실패 요인 기록
- 문서화 단계: 남은 리스크와 후속 과제 정리

즉, 이 문서는 문제를 나중에 찾기 위한 문서가 아니라,
문제를 미리 드러내고 범위를 제어하기 위한 문서다.

---

# 11. 결론

이 템플릿은 단순한 리스크 메모장이 아니다.

이 문서는
- 구현 전에 위험을 드러내고
- 진행 중 판단 근거를 남기고
- 완료 여부를 더 정직하게 평가하고
- 다음 단계의 우선순위를 정하는 데 도움을 주기 위한
하네스 엔지니어링의 핵심 보조 도구다.

`office-unify` 프로젝트에서는
중요한 구조 변경, 기능 이관, 운영 전환 작업마다
가능하면 이 템플릿을 활용해 리스크를 관리한다.
